# AI 特有のアンチパターン

このドキュメントは `[AI-Antipattern]` 観点で AI レビュアが検出すべきパターンを分類する。`perspectives.md` の `[AI-Antipattern]` 観点の詳細カタログ。

## このカタログが必要な理由

AI 生成成果物は **AI 特有の罠** を持つ。これらは人間が書くコードでは出にくく、人間のレビュー観点をそのまま AI レビュアに移植するだけでは拾えない。本フレームワークが既に持っている「**埋め合わせ禁止**」（ハードルール）はそのうちの 1 種類だが、運用していると他のパターンも次々と出てくる。

> AI に対するレビューは、AI 特有の罠を知ったうえで設計する必要がある。
> 人間のレビュー観点をそのまま移植するだけでは出てこない発想。

「典型的な逃げ方を先回りで全部潰す」のがこのカタログの設計思想。新パターンが見つかったら追記する（human-on-the-loop）。

## カタログ構成

各パターンは以下の形式で記述:

- **症状**: AI がどう振る舞うか
- **検出**: どの観点・どの grep で拾うか
- **severity**: Critical / Major / Minor
- **対応**: AI に返すフィードバック

## A. 上流不完備への対応パターン

### A-1. 埋め合わせ進行（既存ハードルール）

**症状**:

- 「ここは XX という前提で進めます」と言って先に進む
- 「SPEC に明示はないが恐らく…」と言って GAP を起票せずに RBA を描き始める
- UC で曖昧な例外フローを RBA が勝手に補う

**検出**:

- 上流成果物に GAP が起票されていないのに下流成果物が完成している
- 下流成果物に「前提として」「恐らく」「一般的に」の語が頻出

**severity**: Critical

**対応**:

> その箇所は SPEC が答えていないので、推測で進めず GAP として起票してください。上流に戻ります。

### A-2. 推測による下流補完

**症状**:

- SPEC で曖昧な責任分担を UC が勝手に決める
- DD でずれた型設計を SRC が勝手に補正する

**検出**:

- SPEC に書かれていない責任分担が UC で初出
- DD に定義されていない型が SRC に登場

**severity**: Critical

**対応**:

> その判断は上流のどの成果物に基づいていますか? 引用元が示せない場合、ADR で記録するか GAP として起票して上流に戻ります。

## B. 降格・先送りパターン

### B-1. 既存違反追随による降格

**症状**:

- 「ただし既存コードも同様に違反しているため、本 PR では Nit とします」
- 「既存パターンに従ったので問題なし」
- 既存違反を正当化根拠にしようとする

**検出**:

- レビュアの severity 判定で「既存」「現状」「従来」が降格理由として登場
- 同じガイドライン違反が複数 PR で連続して Nit 扱いされている

**severity**: Critical（降格を実行したら）/ Major（提案にとどまったら）

**対応**:

> `severity.md` の降格禁止ルールに従い、既存パターン違反を理由に降格しません。既存違反は別 PR で修正対象とし、本 PR の新規追加は元の severity で指摘します。

### B-2. 別 PR / 段階的対応への先送り

**症状**:

- 「別 PR で対応します」
- 「段階的に修正していく方針です」
- 「次のセッションでやります」

**検出**:

- レビュアコメントに「別 PR」「段階的」「次回」が登場
- GAP として起票せずに先送りされた指摘がある

**severity**: Critical（実行したら）/ Major（提案にとどまったら）

**対応**:

> 先送りは禁止です。GAP として正規に起票するか、本 PR で解消するかの 2 択です。

### B-3. TODO/FIXME 残置

**症状**:

- コード内に `// TODO: 後で修正` を残してレイヤ汚染を放置
- DD に `// FIXME: 型設計再考` を残して曖昧さを放置

**検出**:

- `grep -rn "TODO\|FIXME"` で新規追加された TODO/FIXME
- TODO/FIXME 数が PR で純増している

**severity**: Major

**対応**:

> TODO/FIXME での先送りは禁止です。GAP として起票するか、ADR として記録するか、現 PR で解消するかの 3 択です。

## C. AI 生成コード特有の罠（SRC レベル）

### C-1. 幻覚 API

**症状**:

- 存在しないメソッドを呼ぶ（`user.findOrCreate()` のような正しそうだが未定義のメソッド）
- 存在しないライブラリ関数を import
- バージョンに合わない API シグネチャ

**検出**:

- `[Trace]`: SRC 内の外部依存呼び出しが DD で定義された API surface に存在しない
- 言語ツールのコンパイル / 型チェック

**severity**: Critical

**対応**:

> このメソッドは DD の API surface に定義されていません。実在を確認したうえで DD に追加してから SRC に書くか、別の方法で実装してください。

### C-2. エラー握り潰し / フォールバック濫用

**症状**:

- 上流 API が落ちたら無言で空配列を返す
- catch 句で何もせず通過
- エラーを上位に伝播せずデフォルト値で補う

**検出**:

- `catch` ブロックが空、または `return []` / `return null` のみ
- エラー処理が `// ignore` のみのコメント

**severity**: Major

**対応**:

> エラー握り潰しは NFR / SPEC で許可されていない限り禁止です。エラー方針を SPEC §「エラー方針」に従って実装するか、SPEC を更新してください。

### C-3. scope creep（要求外の変更）

**症状**:

- 1 関数の修正依頼で同ファイル全体 reformat
- バグ修正の PR で無関係なリファクタが混ざる
- 「ついでに」見つけた箇所まで触る

**検出**:

- PR の diff 行数が指示内容に対して桁違いに大きい
- 触られたファイルが指示範囲を超える
- コミットメッセージが複数の独立変更を含む

**severity**: Major

**対応**:

> PR のスコープは元の指示範囲に限定してください。発見した別問題は別 PR / 別 GAP で起票してください。

### C-4. dead code 残置

**症状**:

- refactor で旧実装が dead code 化
- 新実装のみで参照される旧関数を残置

**検出**:

- 言語ツールの dead code lint
- 関数定義に対する参照 0 件

**severity**: Minor

**対応**:

> dead code を削除してください。後方互換が必要な場合は ADR で例外として記録してください。

### C-5. 不要な後方互換コード

**症状**:

- 内部関数なのに deprecated alias を新設
- 公開しない API に多重シグネチャを残す

**検出**:

- DD で内部関数指定されているのに deprecated 修飾子付き alias
- 同じ機能の 2 つ以上の関数定義

**severity**: Minor

**対応**:

> 内部関数の後方互換は不要です。alias を削除してください。

## D. 基準書き換えパターン（最重要・メタレベル）

### D-1. 品質基準そのものを緩める変更

**症状**:

- テストカバレッジの閾値を引き下げる PR
- 自作 lint ルールの対象範囲を狭める PR
- ガイドラインの文言を「必須」から「推奨」に書き換える PR
- `.legixy.toml` の chain 順を緩める PR

**検出**:

- PR の diff に以下のパスが含まれる:
  - `review-guidelines/` 配下
  - `docs/SCP/` 配下の規定文書
  - `.legixy.toml` の `[id.chain]` セクション
  - `scripts/trace-check.sh` のチェック削除・条件緩和
  - `docs/perspectives/` 配下の観点削除
  - `bootstrap/CLAUDE*.md.template`
  - ハードルール本文の変更

**severity**: Critical（人間 Approve 必須領域）

**対応**:

> 品質基準を緩める変更は AI レビュアが Approve できません。人間レビュアの判断領域です。緩和が必要な根拠を ADR で記録し、人間レビュアに判断を委ねます。

### D-2. ハードルール緩和の正当化

**症状**:

- 「現実的な運用負荷を考えると…」
- 「既存実装が違反しているので基準を合わせるべき」
- 「AI が間違えやすいので閾値を下げるべき」

**検出**:

- ADR / PR description に「緩和」「現実的」「運用負荷」が登場
- ハードルール 9 スキップ ADR の連続発行（過去 N 件で警告）

**severity**: Critical

**対応**:

> 緩和の正当化として認められない理由です。ハーネスの改善で対処すべき問題で、基準を下げて運用を維持するのは旧プロセスの破綻パターンです。

### D-3. ゲートスキップの常態化

**症状**:

- ゲートスキップ ADR が短期間に複数発行
- 特定 SPEC に対する前段スキップが繰り返される
- スキップ理由が「時間がない」「リリース予定」

**検出**:

- `docs/adr/` 配下の `frontend-pass-skip-*.md` / `gate-skip-*.md` の発行頻度
- スキップ ADR の理由欄に「時間」「リリース」「とりあえず」が登場

**severity**: Major（単発）/ Critical（連続）

**対応**:

> スキップが常態化すると本プロセスは形骸化します。スキップを Major カテゴリの早期警戒指標として monitoring し、頻度が閾値を超えたら根本対応の ADR を起票してください。

## E. AI が自分を守るパターン

### E-1. 自己フィードバックの過剰受容

**症状**:

- AI レビュアが「指摘した自分の判定が間違っていたかも」と修正案を Nit に降格
- author AI が reviewer AI の指摘に「ご指摘の通り Nit に修正します」と過剰応答

**検出**:

- 同一 PR 内で同一指摘が複数回 severity 降格
- レビュアが「失礼しました」「再検討しました」と判定を覆す

**severity**: Major

**対応**:

> AI 同士の自己フィードバックループによる基準緩和は禁止です。判定が割れる場合は人間レビュアに escalate してください。降格禁止ルールが優先します。

### E-2. レビュアとして自分が書いたコードを通そうとする

**症状**:

- 同一 AI セッションが author と reviewer の両方を兼ねる運用
- レビュア起動時のコンテキストに author 視点の前提が残存

**検出**:

- レビュア起動時に CLAUDE.md が `CLAUDE.md.template`（Author モード）のままになっている
- author / reviewer のセッション分離が確認できない

**severity**: Critical

**対応**:

> author と reviewer は別セッション・別 CLAUDE.md（`CLAUDE-reviewer.md.template`）で動かしてください。同一セッションでの兼務は判定の独立性を壊します。

## F. 用語・概念の汚染パターン

### F-1. ドメイン外語彙の流入

**症状**:

- SPEC で定義していない用語が UC / RBA で初出
- 「在庫」概念に「破棄物品」が混入（概念領域の汚染）
- 業界用語と一般用語の混在

**検出**:

- `[Consistency]` の ICONIX 流三者整合性検証
- SPEC の用語集に存在しない単語が下流に出現

**severity**: Major

**対応**:

> SPEC で定義していない用語を下流で使うのは禁止です。SPEC の用語集を更新するか、別の表現に置き換えてください。

### F-2. 言語固有概念の早期混入

**症状**:

- RBA に「`async` 操作」と書く
- SEQA に「`await` を待つ」と書く
- RBD に「`Promise` を返す」と書く

**検出**:

- `[Layer]` のレイヤ汚染検査（`scripts/trace-check.sh` [5/5]）
- 抽象側・具体側に言語キーワードが grep でヒット

**severity**: Critical（具体側 + 言語識別子）/ Major（抽象側 + 概念汚染）

**対応**:

> 言語固有概念は DD で初出のルールです。抽象側 / 具体側ではドメイン語彙 / クラス図表記までに留めてください。

## G. 修正フローでの罠

SCP で追加された `defect-fix` / `spec-change` / `spec-add` イベントフロー (`10-modification-events.md`) で、AI Reviewer が陥る特有の罠。

これらは新規作成中心のフローでは現れず、修正フロー固有のパターンである。Phase E1 〜 E5 (`10-modification-events.md` §10) の各段階で新パターンが発見されると想定されるため、本セクションは継続的に育成される。

### G-1. 浅い読解

**症状**: `defect-fix` Step 4 で AI Reviewer が上流ドキュメント群を表面的にしかチェックしない。章節レベルの記述で「整合している」と判定し、subnode (`#anchor` レベル) の不備を見逃す。

**例**: DD-APP-005 の §3「保存処理」を確認し、概要が SPEC-APP-001 §3.2 と一致しているので APPROVE。しかし `#s:concurrent-write` 配下の競合制御の記述が SPEC では別の方針 (楽観的ロック) を要求しているのに、DD では悲観的ロックで書かれていることを見逃す。

**検出方法**:

- Reviewer subagent の発火時に「subnode レベルまで読む」を明示的に指示する
- `legixy refresh-subnodes --dry-run` で subnode 差分を提示し、それぞれに対する評価を求める

**severity**: Major (本物の不具合を見逃す効果がある)

**対応**:

> 章節レベルだけでなく subnode (`#anchor` / `#s:label`) レベルまで読み込んで、矛盾を探してください。`legixy refresh-subnodes` の出力を参照すること。

### G-2. 過剰修正

**症状**: `defect-fix` Step 4 で AI Reviewer が、本来の不具合と無関係な記述揺れ・誤字・スタイル違反を「不備」として大量に列挙する。Step 5 の選別作業を開発者に押し付け、認知負荷を増やす。

**例**: 不具合「保存ボタンが反応しない」に対して、関連 DD の typo 3 件、コメントの改行スタイル 5 件、用語揺れ 2 件を含めた 25 件の「不備」を報告。本来の原因 (DD-APP-005 §3 のイベントハンドラ抜け) は 25 件中の 1 件として埋もれる。

**検出方法**:

- Reviewer の出力で、検出された不備の数と severity 分布を確認
- Minor / Nit が 5 件以上含まれる場合、不具合との関連性を再評価させる

**severity**: Major (本物の不具合を埋もれさせる effect)

**対応**:

> 当該不具合の症状「<症状>」と、各検出不備がどう因果関係するかを明示してください。因果関係が説明できない不備は別 GAP として分離してください。

### G-3. 修正の依存無視

**症状**: `defect-fix` Step 6 でドキュメント A を修正したが、A に依存する下流 B / C を再生成しないままにする。`impact` の結果を読まずに修正が完了したと宣言する。

**例**: SPEC-APP-001 §3.2 を修正したが、これに依存する UC-APP-005、UC-APP-007 は古い記述のまま。「SPEC を直したので不具合は解消」と Reviewer が APPROVE。実際には UC が古い SPEC に基づく流れになっており、TC は新 SPEC の動作を期待するため、テスト失敗が継続する。

**検出方法**:

- `defect-fix` Step 7 (`impact` 範囲特定) の出力と Step 8 (再構築) の対応を機械検査
- 影響範囲のうち touched されていないノードがあれば警告

**severity**: Critical (修正したつもりが修正されていない、最悪のパターン)

**対応**:

> Step 7 で特定された impact 範囲のうち、まだ再生成されていないノードがあります: <未対応ノードのリスト>。インクリメンタル再構築 (`10-modification-events.md` §5) を完了させてください。

### G-4. 対症療法

**症状**: 不具合の真の原因が上流 (DD / SPEC) にあるのに、SRC を直接 patch して解消したように見せる。Step 6 のドキュメント修正を skip して、SRC を変更するだけで完了させる。

**例**: 「保存処理が遅い」という不具合に対し、DD-APP-008 で定義された「同期保存」を疑わず、SRC レベルでキャッシュを追加。表面的にはパフォーマンスが改善するが、DD の設計判断 (同期保存の意図) との乖離が残り、後に「キャッシュが古い」という別の不具合を誘発する。

**検出方法**:

- `defect-fix` Step 6 で SRC のみが修正され、上流ドキュメント (DD 以上) が touched されていない場合に警告
- commit diff で SRC のみが変更されている `defect-fix` イベントを抽出

**severity**: Major (将来の不具合を誘発する)

**対応**:

> 不具合の原因として SRC レベルでの patch を選択する場合、上流 (DD / SPEC) との乖離を ADR で記録してください。原因が真に SRC レベル (例: typo) であることを Step 4 の検査結果で示すこと。

### G-5. サイクル放置

**症状**: 3 サイクル超過警告 (`scripts/count-fix-cycle.sh`) が発動しても、個別修正を継続する。設計レベルの見直しを行わず、修正を重ねるたびに新しい不整合を誘発する。

**例**: defect-fix イベント 1 → spec-change 2 → defect-fix 3 → spec-change 4 → ... と進む中で、毎サイクル新たな不整合が見つかる。「あと 1 サイクルで終わる」を繰り返しながら 6 〜 7 サイクル目に到達する。

**検出方法**:

- `.scp/event-log.txt` でサイクル数 4 以上のエントリが存在
- かつ、それに対応する ADR (設計見直し記録) が `docs/adr/` に存在しない

**severity**: Critical (修正フローが収束せず、設計の根本問題を隠蔽している)

**対応**:

> 3 サイクル超過の警告を受けても継続する場合、必ず ADR を起票してください。継続理由が明確でない場合は、現在のサイクルを停止し、これまでの修正履歴 (`event-log.txt`) を点検して設計レベルの見直しを行ってください。

## H. 人間関与境界の侵食パターン (ハードルール 11)

SCP で導入された **ハードルール 11(人間関与は SPEC と UC に限定)** の遵守を妨げるパターン。確率論的変換器を AI コンパイラとして使うとき、人間関与の集中を維持できないと、`00-philosophy.md` §3.1 で示した防御線全体の効率が損なわれる。

### H-1. 人間関与の下流への誘発

**症状**:

- AI が下流(RBA, SEQA, RBD, SEQD, DD)の生成中に「ここは人間判断が必要ではないですか?」と過剰に確認を求める
- AI が下流の些細な選択について「念のため確認してください」と人間に問い合わせる
- AI Reviewer が下流ゲートで `NEEDS_HUMAN` を頻発させる(本来 `APPROVE` 可能な領域で)

**検出**:

- 下流自律実行領域(RBA → SRC)のゲートで `NEEDS_HUMAN` の VERDICT が出ている
- AI Author が下流生成中に「確認してください」「判断をお願いします」を頻発
- `.scp/verdict.log` で下流ゲートの `NEEDS_HUMAN` 率が上昇傾向

**severity**: Major(人間関与境界の構造的崩壊につながる)

**対応**:

> 下流(RBA 以降)は AI 自律実行領域です。`08-gates.md` の Approve 権限表に従って、APPROVE / REQUEST_CHANGES のいずれかを選んでください。下流で人間判断が必要と感じた場合、それは多くの場合、上流(SPEC または UC)に GAP として遡って起票するべきパターンです。下流で人間判断を要求する前に、GAP[UC] / GAP[SPEC] 起票を検討してください。

### H-2. 上流変更の下流での「埋め合わせ」

**症状**:

- 下流生成中に SPEC / UC との乖離を発見したが、`/spec-change` イベントを発火せず、下流で辻褄を合わせる
- UC の曖昧性を RBA レベルで「合理的に解釈」して進める
- DD で SPEC と異なる契約を「実装の都合」で定義する

**検出**:

- `defect-fix` Step 4 の Reviewer 検査で、下流が SPEC/UC と乖離している
- commit diff で下流のみが変更され、対応する SPEC/UC 更新が伴わない
- ADR が起票されないまま、下流の設計判断が累積する

**severity**: Critical(ハードルール 11 違反、`00-philosophy.md` §3.1 根拠 3 の構造を破壊)

**対応**:

> 上流との乖離を発見した時点で、`/spec-change` または `/defect-fix` イベントを発火してください。下流での「埋め合わせ」は禁止されています(`00-philosophy.md` §2.1、ハードルール 11)。下流のみの修正で進めると、不具合が表面化しないまま catastrophic failure リスクが累積します。

### H-3. 境界 API 凍結リストの肥大化

**症状**:

- AI が DD 確定時に提示する「凍結対象 API リスト」が、人間がレビュー不可能な規模に膨らむ
- 凍結対象 API に内部実装の詳細(非 pub 関数)が混入する
- リスト提示時の説明が長大で、人間が一括承認の判断ができない

**検出**:

- 凍結対象 API リストが 1 UC あたり 50 件超
- 凍結対象に `pub(crate)` や private なものが含まれる
- リスト本体の説明文が 500 行を超える

**severity**: Major(ハードルール 11 §1.3 例外の運用条件が崩れる)

**対応**:

> 凍結対象 API は外部公開境界(クレートの pub re-export 到達可能なもの)に限定してください。内部実装は凍結対象外です。リストはシグネチャ + エラー型 + 冪等性 + スレッド契約 + バージョン互換性ポリシーのみを 1 API あたり 1 行で記述し、人間が一括承認可能な形式にしてください。

### H-4. AT スキップによる分布外側観察の喪失

**症状**:

- 「機能テストが全て pass しているのでリリース可能」と AT を skip する
- AT 失敗を「軽微な UX 問題」として GAP 化せず放置
- AT の実施頻度をマイルストーン到達時から「リリース前のみ」に減らす

**検出**:

- リリース直前で AT の実施履歴が空または極めて少ない
- `07-at-and-nfr.md` の perspectives.md 昇格機構が機能していない(新観点が長期間追加されていない)
- AT 結果を経由しない直接的なリリース判断が増えている

**severity**: Critical(ハードルール 11 と `00-philosophy.md` §2.4 の分布外側観察が機能しない)

**対応**:

> AT は確率分布外側からの観察として、人間関与境界の重要な一部です。AT が機能しないと、AI Reviewer が見落とした左裾サンプルを検出する経路が失われます。マイルストーン到達時の AT 実施は省略不可です。

### H-5. 人間関与境界の「効率最適化」要求

**症状**:

- 「下流は AI に任せるなら、上流の SPEC/UC レビューも軽くして良いのでは?」という主張
- QSET 質問への回答を AI 自動回答に置き換える提案
- UC のフロー妥当性レビューを Reviewer subagent に委譲する提案

**検出**:

- ハードルール 1, 11 の緩和を求める ADR や PR
- 人間関与作業の自動化を提案するセッション

**severity**: Critical(`00-philosophy.md` §3.1 の構造的根拠を毀損する、`ai-antipattern.md` D-1, D-2 と密接に関連)

**対応**:

> 人間関与境界(SPEC + UC + AT)は「効率」で評価する対象ではなく、「確率論的変換器の外側からの観察基準」として構造的に必要です(`00-philosophy.md` §3.1)。これを自動化することは、AI Reviewer 層の同質モデル盲点を防ぐ最後の独立経路を失うことを意味します。緩和提案は人間 Approve 必須領域として `REQUEST_CHANGES` を返してください。

## カタログの育成

新パターンが見つかったら、以下の手順で追記:

1. **症状** を具体的に記述（実例があれば pseudo code として残す）
2. **検出方法** を機械化可能なら grep / lint コマンドで、不可なら観点として記述
3. **severity** を `severity.md` の基準に従って分類
4. **対応** として AI に返すフィードバックの定型文を用意

カタログ追加の判断は人間レビュアの責務（`severity.md` §3 の人間 Approve 必須領域）。
