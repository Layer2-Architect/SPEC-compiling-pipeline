# 03 — 仕様レベル TDD ループ（SPEC ⇄ TP ⇄ GAP, UC ⇄ TP ⇄ GAP）

このドキュメントは SPEC, UC, TP, GAP の作業時に参照する。

> **前提**: 本ループに着手する SPEC は **前段ループ（`03a-frontend-pass.md`）で FCR.frontend_status = ACCEPTED に到達済** であること。前段未完了の SPEC に対して TP[SPEC] を作るのはハードルール 9 違反。`scripts/trace-check.sh` がこれを機械検証する。

## 1. ループの全体像

```
SPEC（ドラフト）
  ↓
TP[SPEC] 生成 ─────────► 観点ナレッジベース参照
  ↓                       (docs/perspectives/core-perspectives.md など)
GAP[SPEC] 分析
  ↓
SPEC が答えられない観点があれば → SPEC 修正 → ループ先頭に戻る
SPEC が全観点に答えれば → GREEN → UC 生成へ進む
```

UC レベルでも同型のループを回す。SPEC が GREEN になるまで UC に進まない。UC が GREEN になるまで RB に進まない。

## 2. SPEC の構造（推奨）

SPEC は以下のセクションを含む（テンプレ: `templates/SPEC-template.md`）:

- **背景・動機**（Why）
- **対象スコープ**（What）
- **責任分担**（誰が何を保証するか。AI / 人間 / システム / 外部依存）
- **不変条件**（変わらない性質）
- **境界条件**（許容される入力範囲・状態範囲）
- **エラー方針**（部分成功・ロールバック・リトライ）
- **成功の定義**（観察可能な事後条件）
- **変更ポリシー**（誰の承認で変えられるか）

「成功の定義」は **観察可能化** されている必要がある（外部から見て成功/失敗が判定できる）。観察不能な「内部のうまく動いている感」を成功条件にしない。

## 3. TP の生成手順

### Step 1: 観点カテゴリの列挙

少なくとも以下を網羅する:

- **境界値**: 数値・サイズ・時間の上限/下限/0/負/最大/最大+1
- **エラーハンドリング**: 入力不正、外部障害、リソース枯渇、タイムアウト
- **権限・状態**: 操作可能性、ロック、認可、状態遷移の妥当性
- **競合**: 並行操作、Undo 中の操作、外部更新中の操作
- **外部連携**: 通信失敗、部分成功、不整合な応答、リトライ可否
- **ライフサイクル**: 初期状態、空状態、最後の 1 件、削除可否
- **バージョニング**: 履歴・分岐・マージ・互換性

### Step 2: 領域固有観点の追加

`docs/perspectives/` 配下のナレッジベースを参照し、領域固有の観点を追加する。汎用観点だけでは領域の核心を外す。

### Step 3: 観点を仕様に「ぶつける」

各観点について、SPEC が以下のいずれかを満たすか確認:

- 観点に対する明示的な答えが SPEC にある → その観点は GREEN
- SPEC に答えが無い、または曖昧 → その観点は RED、GAP 生成対象

### Step 4: TP の記録（テンプレ: `templates/TP-template.md`）

```markdown
Document ID: TP-<AREA>-001

# TP-<AREA>-001: <タイトル>

**親 SPEC**: SPEC-<AREA>-NNN
**ステータス**: red

## 対象スコープ
<このTPがカバーする SPEC の章節>

## 観点リスト

### 境界値
- [ ] 観点 1
- [ ] 観点 2

### エラーハンドリング
- [ ] 観点 3
...

## RED / GREEN 判定
| 観点 | 判定 | 関連 GAP |
|---|---|---|
| 観点 1 | RED | GAP-<AREA>-001 |
| 観点 2 | GREEN | (SPEC §X.Y で回答済) |
```

**重要**: TP は「テストケース」ではなく「観点」。具体的なテストデータや期待値ではなく、「何を確認すべきか」のリスト。具体化（テストデータ・期待値）は TS 層で行う。

## 4. GAP の記録（テンプレ: `templates/GAP-template.md`）

各 RED 観点について GAP を生成:

```markdown
Document ID: GAP-<AREA>-001

# GAP-<AREA>-001

**親 TP**: TP-<AREA>-001
**観点**: <TP のどの観点か>
**現状の SPEC**: SPEC-<AREA>-NNN §X.Y で <現状> に触れているが、<不足> は未定義
**期待される情報**: <何が SPEC に追加されるべきか>
**影響範囲**: <この GAP が close されないと UC や下流に与える影響>
**ステータス**: open
```

## 5. GAP のクローズ手順

SPEC が修正され、該当観点に答えるようになったら:

1. SPEC 修正のコミットで GAP のステータスを `closed (YYYY-MM-DD)` に更新
2. GAP 本文に「解決経緯」を簡潔に記録（どの SPEC §X.Y で何を追加したか）
3. close 済み GAP は `docs/gap-analysis/` に保持（archive 移動はしない。graph.toml に登録済のため）
4. 関連 TP のステータスを再評価。全観点が GREEN になったら TP のステータスを `green` に変更

**全 GAP が close されるまで UC 生成に進まない。** `bash scripts/trace-check.sh` がこれを検証する。

## 6. UC レベルでも同じループを回す

UC が確定したあと、UC 用の TP / GAP ループを回す。UC レベルの観点は SPEC レベルとは異なる:

### UC レベル固有観点

- **基本フロー**: 各ステップが先行ステップの事後条件を満たすか
- **代替フロー**: 分岐条件が網羅されているか
- **例外フロー**: 各ステップでの失敗パスが定義されているか
- **アクター遷移**: アクターの権限・状態変化が一貫しているか
- **データフロー**: 入出力データの型・制約・ライフタイム

これらは SPEC レベルでは抽象的にしか書けない。UC まで具体化されて初めて観点として成立する。

## 7. UC の構造（推奨）

UC は以下のセクションを含む（テンプレ: `templates/UC-template.md`）:

- **ID と名称**
- **親 SPEC**（参照）
- **アクター**（主要アクター・副次アクター）
- **前提条件**
- **基本フロー**（番号付きステップ）
- **代替フロー**（基本フローのどのステップから分岐するか）
- **例外フロー**（各ステップでの失敗パス）
- **事後条件**（成功時 / 失敗時）
- **関連 SPEC §**

## 8. SPEC 変更時のリグレッション

SPEC を変更したら、変更箇所に関連する全 TP を再評価する。legixy の semantic check が drift を検出するが、これは出発点であり、人間と AI が協働で TP の妥当性を再確認する。

```bash
# 影響範囲の特定
legixy impact SPEC-<AREA>-NNN
```

`impact` が報告する全成果物について、第 2 層 semantic check が pass することを確認。pass しないものは drift しているので追従修正する。

## 9. AI と人間の分担

| 作業 | AI 主体 | 人間主体 |
|---|---|---|
| TP の初稿生成 | ◯ | レビュー |
| 観点ナレッジベース参照 | ◯ | 領域固有観点の追加 |
| GAP の検出・起票 | ◯ | 重要度判断 |
| SPEC 修正案の作成 | ◯（提案） | **承認・確定**（ハードルール 1） |
| GAP のクローズ判定 | ◯（候補提示） | 確認 |
| UC ドラフト | ◯ | フロー妥当性レビュー |

**SPEC の最終確定は必ず人間。** AI は提案・分析・差分提示までで止まる。

## 10. ループが収束しないケース

TP を増やしても GAP が増え続ける場合、SPEC のスコープが広すぎる兆候。SPEC を分割するか、責任分担を別 SPEC に外出しする。

GAP が同じ箇所に集中する場合、SPEC のその章節が **論理的に矛盾している** 可能性がある（互いに排他的な要件が並立している等）。SPEC を改訂する。

## 11. ループの終了判定（GREEN）

以下を全て満たすこと:

- 全 TP のステータスが `green`
- 全 GAP のステータスが `closed (YYYY-MM-DD)`
- `bash scripts/trace-check.sh` が pass
- TP が観点ナレッジベース（汎用 + 領域固有）の主要カテゴリを網羅している（人間レビュー）

GREEN になったら次のフェーズ（UC レベルなら RB、SPEC レベルなら UC）に進む。
