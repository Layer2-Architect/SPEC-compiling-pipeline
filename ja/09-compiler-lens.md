## §1 SCP の基本構造

### 1.1 パイプラインの定義

SCP は以下の直列パイプラインで構成される(→ `01-overview.md`)。

```
SPEC → UC → RB → SEQ → DD → TS → TC[RED] → SRC → TC[GREEN]
```

| 段階 | 役割 |
|---|---|
| SPEC | 真のソース。人間が作成・承認する唯一の保持対象 |
| UC | SPEC から抽出される利用シナリオ |
| RB | UC を主語・責務・境界に分解した空間構造 |
| SEQ | RB を時系列的な相互作用として展開 |
| DD | 実装直前の構造定義(クラス図相当) |
| TS / TC | DD から導出される検証定義 |
| SRC | 採用言語による実装 |

### 1.2 中間ファイルとしての UC 以降

SCP の設計原則として、SPEC のみが真のソースであり、UC 以降のすべての成果物は中間ファイルとして扱われる。SPEC が改訂された場合、中間ファイルは再生成される対象である。

中間ファイルとして扱うことで各段階の決定が永続的コミットメントにならない。後の知見で各段階の判断を変更しても、SPEC の整合性が保たれていれば再生成で対応できる。

ただし「再生成」が何を意味するかは §8 で精緻化する。日常運用ではフル再生成ではなく、legixy 有向グラフを用いた局所再生成が主たる経路となる。

### 1.3 役割分担エージェント

SCP は VSDD の役割分担エージェントを使用する(詳細は `guides/ai-collaboration.md`)。

| 役割 | 責務 |
|---|---|
| designer | 各段階での降下作業(UC→RB→SEQ→DD など) |
| implementer | SRC の実装 |
| qa-runner | 機械的検証の実行 |
| Adversary | 意味的検証、designer 出力への異議申し立て |

### 1.4 legixy の三側面

各段階間の依存関係を有向グラフとして記録する追跡装置。SPEC から各中間ファイル、SRC までの依存を `graph.toml` に保持し、SPEC 起源の完全性を検証する。

本書では legixy を **§3 でシンボルテーブル/DWARF**、**§7 でライフサイクル全体の中央装置**、**§8 でインクリメンタル・コンパイラの依存グラフ** として、異なる側面から記述する。三つは矛盾せず、参照する側面が異なる。

## §2 前段ループ(フロントエンド・パス)

SPEC 駆動が成立するためには、SPEC が形式的に完備している必要がある。しかし人間が初稿として書いた SPEC は必ず不完備である。前段ループは、この不完備を解消し、後段のコンパイル可能な状態へ SPEC を正規化するフロントエンド機構である。

> **SCP における位置付け**: 本書 §2 の内容は SCP のプロセス本体に **`03a-frontend-pass.md`** として実装規定化されており、ハードルール 9 として「SPEC は前段ループで FCR.frontend_status = ACCEPTED に到達していなければ TP[SPEC] / UC 着手禁止」が機械検証される。本書 §2 はその構造的位置付け(コンパイラのフロントエンド・パスとの対応)を提供する視点であり、運用手順は `03a-frontend-pass.md` を参照する。
>
> SPEC レベル TDD ループ(SPEC ⇄ TP ⇄ GAP, → `03-spec-level-tdd.md`)との関係: 前段ループが先(コンパイル必須項目の充足検査)、SPEC レベル TDD が後(網羅性確認)。前段ループ ACCEPTED が TP[SPEC] 着手の前提条件。

### 2.1 前段の責務

**前段とは**、自然言語 SPEC に対して AI が必要項目テンプレートを仮充填し、不足・矛盾・曖昧性を検出し、開発者への質問票として返し、回答を反映しながら、コンパイルに必須な内容が欠落していない状態へ到達させる処理である。

この処理は AI に設計判断を委任するものではない。AI は開発者の入力をもとに、後段変換に必要な情報が揃っているかを確認するフロントエンド検査器として振る舞う。

### 2.2 全体フロー

```
Raw Natural Language SPEC
 → AI による必要項目テンプレート仮充填
 → AI による不足・曖昧性・矛盾・境界不明の検出
 → 開発者への質問票生成
 → 開発者による回答
 → AI による SPEC 差分案作成
 → 開発者による差分承認
 → テンプレート再充填
 → フロントエンド検証
 → 未解決項目があれば反復
 → Accepted SPEC
 → UC 以降の SCP 本体へ降下
```

前段ループは「質問が出なくなるまで会話する」処理ではない。AI が内部で必要項目テンプレートを埋め、後段コンパイルに必須な項目について未定義・矛盾・境界不明・例外未処理が検出されなくなるまで、質問と回答を反復する処理である。

### 2.3 人間に見せる操作

SCP 前段は人間に AST、IR、用語表、状態遷移表、検査項目一覧を直接記述させない。人間に見せる操作は、自然言語による説明、質問への回答、AI が作成した SPEC 差分の承認に限定する。

| 人間に見せるもの | 目的 |
|---|---|
| 初稿 SPEC 入力欄 | 作りたいもの、振る舞い、制約、期待結果を自然言語で入力する |
| 質問票 | AI が不足・曖昧・矛盾と判定した点について回答する |
| 選択肢 | 複数の解釈候補がある場合、人間が意図に近いものを選ぶ |
| SPEC 差分案 | 回答を反映した SPEC 変更内容を確認する |
| 承認操作 | 前段確定、または追加修正を判断する |

**人間に要求しないこと**: 必要項目テンプレートを手で埋めること / 構文木・IR・状態遷移表を理解すること / UC 以降の中間ファイルを直接編集すること / AI 内部の検査パスを意識すること。

### 2.4 AI が内部で行う処理

| 内部処理 | 役割 | 出力 |
|---|---|---|
| 必要項目テンプレート仮充填 | SPEC から機能、非機能、制約、例外、外部 I/O などを抽出し、後段コンパイルに必要な項目へ仮配置 | Filled Template Draft |
| 用語抽出 | ドメイン語、役割名、状態名、イベント名、外部システム名を抽出 | Hidden Glossary |
| 主語・責務抽出 | 誰が、何に対して、何を行うかを抽出 | Responsibility Candidates |
| 境界抽出 | システム内外、責務範囲、外部依存、ユーザー責任範囲を抽出 | Boundary Candidates |
| 不足検出 | 後段変換に必要だが SPEC に明示されていない情報を探す | Missing Item List |
| 曖昧性検出 | 複数解釈可能な語、条件、状態、責務範囲を検出 | Ambiguity List |
| 矛盾検出 | 同一対象に対する競合する要求、状態、制約を検出 | Contradiction List |
| 例外・失敗時挙動検出 | 失敗、タイムアウト、権限不足、外部 API 失敗などの未定義挙動を検出 | Exception Gap List |
| 質問票生成 | 検出結果を人間が回答可能な質問に変換 | Questionnaire |
| 回答反映 | 開発者の回答を SPEC 差分案へ変換 | SPEC Patch Proposal |
| 検証 | テンプレート充足、参照整合性、矛盾不在、コンパイル必須項目充足を確認 | Frontend Check Result |

### 2.5 必要項目テンプレート

必要項目テンプレートは人間が直接編集するフォームではない。AI が自然言語 SPEC から仮充填し、不足項目を質問票へ変換するための内部構造である。

| 項目群 | 内容 | 未充足時の扱い |
|---|---|---|
| 目的 | システムが何を達成するためのものか | 目的質問 |
| 利用者・外部主体 | ユーザー種別、外部システム、管理者、ゲストなど | 主体質問 |
| 主要機能 | 観測可能な機能、ユースケース候補 | 機能質問 |
| 入力 | ユーザー入力、外部 API 入力、ファイル、イベント | 入力質問 |
| 出力 | 画面表示、保存結果、通知、API 応答、生成物 | 出力質問 |
| 状態 | ライフサイクル、状態遷移、保存状態、一時状態 | 状態質問 |
| 例外・失敗時挙動 | 異常系、外部依存失敗、権限不足、タイムアウト、重複、欠損 | 例外質問 |
| 制約 | 技術制約、運用制約、法令、セキュリティ、データ保持 | 制約質問 |
| 非機能要件 | 性能、可用性、保守性、監査性、可観測性、移植性 | 非機能質問 |
| 受け入れ条件 | どの状態になれば要求を満たしたと判定するか | 完了条件質問 |
| 禁止事項 | してはいけない挙動、許可しない状態、避ける実装方針 | 禁止事項質問 |

テンプレートは人間用入力フォームではなく、AI frontend が自然言語 SPEC を後段変換可能な構造へ正規化するための内部検査表である。人間に見せるのは、テンプレートそのものではなく、テンプレート未充足から生成された質問である。

### 2.6 質問票生成

AI は内部検査結果をそのまま人間へ提示しない。開発者が回答可能な自然言語の質問票へ変換する。

| 内部検出 | 質問票の形式 | 例 |
|---|---|---|
| 未定義語 | 用語確認質問 | 「ここでいう『管理者』はシステム管理者ですか、組織の管理者ですか?」 |
| 複数解釈 | 選択肢付き質問 | 「削除は論理削除ですか、物理削除ですか?」 |
| 例外未定義 | 失敗時挙動質問 | 「外部 API が失敗した場合、再試行/エラー表示/処理保留のどれですか?」 |
| 境界不明 | 責務範囲質問 | 「通知文面の生成は本システムの責務ですか、外部サービスの責務ですか?」 |
| 矛盾 | 衝突解消質問 | 「SPEC-12 では即時反映、SPEC-31 では承認後反映とあります。どちらを採用しますか?」 |
| 非機能不足 | 運用条件質問 | 「同時利用者数、応答時間、ログ保存期間について希望値はありますか?」 |

**禁止事項**:

- AI が勝手に仕様を確定してはならない
- 複数の設計方針を、根拠なく一つに決めてはならない
- 人間に内部テンプレートの全項目を埋めさせてはならない
- 質問は後段コンパイルに必要な不足・曖昧性・矛盾に結びついていなければならない

### 2.7 フロントエンド検証

前段確定前に、AI は内部テンプレートと SPEC 差分をもとに検証を行う。これは Clang における前処理、構文・意味チェック、エラー・警告生成に相当する。

| 検証項目 | 内容 | 失敗時 |
|---|---|---|
| 必須項目充足 | 後段コンパイルに必要な項目が埋まっているか | 質問票へ戻す |
| 用語一貫性 | 同一語が複数意味で使われていないか | 用語確認質問 |
| 主体一貫性 | 同じ主体の責務が矛盾していないか | 責務確認質問 |
| 状態遷移充足 | 主要状態に遷移元、遷移先、異常系が存在するか | 状態質問 |
| 例外経路充足 | 失敗時、権限不足、外部依存失敗が未定義でないか | 例外質問 |
| 境界整合性 | システム内外、外部依存、人間作業の境界が不明でないか | 境界質問 |
| 矛盾不在 | 要求同士が競合していないか | 衝突解消質問 |
| UC 生成可能性 | SPEC から UC 候補を生成できる程度に閉じているか | 不足質問 |

SCP 前段における質問票は単なるヒアリング項目ではなく、**後段変換を阻害する不足・曖昧性・矛盾を人間が修正可能な形で返すコンパイルエラーまたは警告** である。

### 2.8 確定条件

前段は AI が質問を思いつかなくなった時点で確定するのではない。後段コンパイルに必須な項目について検証上の未充足が検出されない状態になり、開発者が SPEC 差分を承認した時点で確定する。

```
if required_template_complete
   and glossary_consistent
   and no_blocking_ambiguity
   and no_blocking_contradiction
   and exception_paths_sufficient
   and boundary_sufficient
   and usecase_generation_possible
   and human_approved:
       frontend_status = ACCEPTED
else:
       frontend_status = NEEDS_QUESTIONNAIRE
```

### 2.9 前段ループが保証することと保証しないこと

**保証すること**:

- 後段コンパイルに必須な情報の未充足を検出し、質問として返す
- 検出可能な矛盾、曖昧性、境界不明、例外未定義を減らす
- 開発者が承認した SPEC 差分だけを確定 SPEC へ反映する
- UC 以降へ降下できる程度に SPEC を閉じる

**保証しないこと**:

- 開発者の真の意図を AI が完全に理解していること
- 承認された SPEC が市場・利用者・組織にとって正しいこと
- 生成されるアプリが人間の期待を完全に満たすこと
- 将来の要求変更が発生しないこと
- AI が検出すべき全ての不足・曖昧性・矛盾を実際に検出すること

前段ループは **形式性のゲートであり、意図性のゲートではない**。意図性の判定は SRC 生成後の人間確認に位置する。また、形式性ゲートそのものも AI の検出能力に依存する近似であり、完全性ではなく到達度として扱う必要がある。

## §3 コンパイラ構造との対応

### 3.1 全体対応表

前段ループを追加した SCP は、伝統的コンパイラと以下の対応関係を持つ。

| コンパイラ | SCP | 備考 |
|---|---|---|
| ソース言語 | SPEC | 人間が記述する唯一の真のソース |
| 字句解析 + 構文解析 + 意味解析(フロントエンド) | 前段ループ | 不完備な入力を形式的に閉じた状態に変換 |
| 多段中間表現(IR) | UC, RB, SEQ, DD, TS, TC | 各段階が異なる抽象度の IR |
| 最適化パス・静的解析パス | VSDD の役割分担エージェント、legixy 検証 | 各段間で機械的・意味的検証を挟む |
| バックエンド(IR → ターゲット ISA) | DD → SRC の言語依存変換 | 採用言語固有の制約導入 |
| ターゲット ISA | 採用言語(Rust, C++, TypeScript など) | 多様な候補から選択 |
| シンボルテーブル・依存グラフ | legixy の graph.toml | 段間の依存と意味的整合性の追跡 |
| ソースマップ・DWARF デバッグ情報 | legixy の逆引き機能 | SRC から SPEC への上流追跡 |
| コンパイルエラー | 段間検証の不通過 | 次段に進めない構造 |

### 3.2 対応の構造的根拠

両者の対応は表面的な類似ではなく、共通の構造原理に基づく。

- **段階的変換**: 入力を単一の変換で出力にせず、複数の中間段階を経由する
- **抽象度の階層化**: 各段階が異なる抽象度を扱う
- **責務分離**: 各段階が独立した責務を持ち、互いに干渉しない
- **検証段階の挿入**: 段間で機械的・意味的検証を行う
- **形式性ゲートと意図性ゲートの分離**: 形式的な閉性と意図的な正しさを別軸で扱う
- **単一情報源**: 真のソースを一つに固定し、他は再生成可能な中間物とする

これらの原理はコンパイラ理論の固有の発明ではなく、複雑な変換問題に対する一般解として独立に到達可能である。

### 3.3 対応の強度

本書では「対応」を以下の強度区分で扱う。すべてを「同型」と一括しないのは、各箇所での主張強度を正確にするためである。

| 強度 | 意味 | 例 |
|---|---|---|
| 構造的対応 | 段階数・順序・各段の責務が一対一で対応する | SPEC ↔ ソース言語、UC 以降 ↔ IR 階層 |
| 機能的対応 | 担う役割は対応するが、実装機構が異なる | legixy graph.toml ↔ シンボルテーブル+DWARF |
| 構造的拡張 | コンパイラ概念を保持しつつ、新しい次元を追加 | 多対多依存グラフ(§7.3) |
| 新カテゴリへの分岐 | 元概念から逸脱し、新カテゴリで再定義 | 非決定論的フロントエンド(§9.2) |

## §4 中段の多段化(分析層と設計層の分離)

> **SCP における位置付け**: 本書 §4 の内容は SCP のプロセス本体に **`04-iconix-layer.md`** として実装規定化されている。RBA(抽象 RB)/SEQA(抽象 SEQ)/RBD(具体 RB)/SEQD(具体 SEQ)/DD の 5 段階として typecode 化され、ハードルール 10 「ICONIX 二段化レイヤ汚染禁止 + 三者整合性検証必須」が機械検証(レイヤ汚染部分)+ Adversary 検査(三者整合性部分)される。本書 §4 はその構造的位置付け(MLIR 多段方言との対応)を提供する視点であり、運用手順は `04-iconix-layer.md` を参照する。
>
> なお、SCP では「抽象側はドメインレベル(UC から導かれる主語・責務、クラス境界なし)」「具体側はクラス図レベル(言語非依存、操作名は人間の言語まで、関数名は除外)」「DD は言語仕様・動作環境・OS API への翻訳」と境界が確定している。これは ICONIX 古典慣習との整合性を取りつつ、本書 §4 の構造原理を保つ設計。Rust / C / Scala / Go / Ruby / Pascal / Python など複数言語で具体側まで共通化が可能。
>
> 抽象層の検証は **Jacobson OOSE 1992 流(UC ⇄ RBA ⇄ SEQA、動的構造整合)** と **ICONIX 2007 流(UC ⇄ RBA ⇄ SPEC、静的概念整合)** の両流派を統合する。両者は補完的で、抽象層 GREEN 確定に同時適用される(`04-iconix-layer.md` §11)。

### 4.1 単段中間 IR の限界

UC → RB → SEQ → DD の流れを単段として扱うと、抽象度の異なる責務が一段に押し込まれる。具体的には、リソース無限大の前提での分析作業と、環境制約下での設計作業が、同じ段で混在する。

### 4.2 二段化された中段

ICONIX の分析/設計分離原則に従い、中段を以下のように分離する。

```
分析層: 抽象RB + 抽象SEQ
  ├─ 主語・責務分解
  ├─ リソース無限大の前提
  └─ 環境制約なし

設計層: 詳細RB + 詳細SEQ
  ├─ 実現可能性の確認
  ├─ クラス図相当の粒度
  └─ 環境制約の導入

DD: クラス図(言語依存)
```

### 4.3 段間境界の判定基準

| 層 | 完了判定基準 |
|---|---|
| 分析層 | UC の全要素が主語+責務に分解されている(完備性) |
| 設計層 | (1) 環境制約下で実現可能性が確認される<br>(2) クラス図相当の粒度まで細分化されている |
| DD | 言語固有の制約が反映され、SRC 生成が機械的に進む状態 |

### 4.4 MLIR との対応

LLVM が単段 IR から多段 IR(MLIR)に移行した理由と、SCP の中段を多段化する理由は構造的に同じである。

| MLIR | SCP |
|---|---|
| 高レベル方言(linalg, tensor 等、ターゲット非依存) | 分析層(リソース無限大、環境非依存) |
| 中レベル方言(affine, scf 等、構造保持しつつ具体化) | 設計層(環境制約導入、言語非依存) |
| 低レベル方言(LLVM IR, GPU 方言等、ターゲット依存) | DD(言語依存) |

### 4.5 マルチバックエンドの分岐点

分析層と設計層が共に言語非依存であれば、マルチバックエンドの分岐点は DD 直前に置かれる。

```
抽象RB/SEQ ─┬─→ 詳細RB/SEQ (Rust向け)    → DD (Rust)    → SRC (Rust)
            ├─→ 詳細RB/SEQ (Scala向け)   → DD (Scala)   → SRC (Scala)
            └─→ 詳細RB/SEQ (TypeScript向け) → DD (TS)   → SRC (TS)
```

分析層は一度書けば全ターゲットで共用できる。これは LLVM のフロントエンド/バックエンド分離と構造的に対応する。なお、この機能は §8 の二モード分離では **フル再生成モード** に属する。

## §5 機能と責務の分離による言語別配置

### 5.1 機能と責務の区別

- **機能**: 言語仕様に内在し、プロセス側に移動不可能な能力(例: Thread、64bit ポインタ、所有権モデル、GC)
- **責務**: 移動可能な機能。担い手を選択できる(例: D 判定、E 判定、SPEC 起源完全性検証、機械的検証)

**判定方法**: ある能力 X をプロセス側で実現できるなら責務、不可能なら機能。

### 5.2 SEQ→DD 変換の 5 操作

SEQ→DD 変換は単一操作ではなく、5 種類の操作の合成である。

| 操作 | 内容 | 距離 | 機械検証 |
|---|---|---|---|
| A | メソッド集約 | 短 | 可能 |
| B | インスタンス→クラスのリフト | 短 | 可能 |
| C | メソッドシグネチャ化 | 中 | 部分可能 |
| D | 関連の意味付け(composition / aggregation / カーディナリティ) | 長 | 不可能 |
| E | 構造的抽象の導入(インターフェース抽出、継承、パターン適用) | 最長 | 不可能 |

### 5.3 E の二分

| E 区分 | 内容 | 判定述語 |
|---|---|---|
| E 前半 | 継承使用判断、インターフェース必要性判断 | 現在時制(コード状態から判定可能) |
| E 後半 | 抽象化タイミング判断、抽象化粒度判断 | 未来時制(将来の変化パターンに依存) |

### 5.4 言語別の責務配置

**Rust 版**:

| 責務 | 担い手 |
|---|---|
| D 判定 | コンパイラ(借用チェッカー、所有権モデル) |
| E 前半 | コンパイラ(半分: 継承消去) + Adversary(半分: トレイト抽出判断) |
| E 後半 | SPEC 駆動で発生抑止(§6 参照) |
| SPEC 起源完全性 | legixy |

**C++ / その他継承可能・所有権非明示言語版**:

| 責務 | 担い手 |
|---|---|
| D 判定 | ICONIX 反復 + Adversary |
| E 前半 | ICONIX 反復 + Adversary |
| E 後半 | SPEC 駆動で発生抑止 |
| SPEC 起源完全性 | legixy |

### 5.5 言語選択の評価手続き

1. UC から機能要件を抽出
2. ターゲット言語の機能と照合(機能要件充足チェック)
3. 充足しなければ別言語選択、または UC 要件緩和
4. 充足したら、プロセスが要求する責務集合を列挙
5. 言語の機能から導出される責務を言語側に配置
6. 残りの責務をプロセス側に配置
7. 配置結果がその言語に対する SCP 具体形

これはコンパイラのターゲット選択時のチェック(ターゲット ISA が必要命令をサポートするか)と構造的に対応する。

## §6 SPEC 駆動による未来予測判断の構造的消去

### 6.1 E 後半が困難である構造的理由

E 後半(抽象化のタイミング判断、粒度判断)は、判定述語が未来時制を要求する。「この抽象は将来必要になるか」は書いた時点では決定不能である。

判定基準が現在時制の事実から構成できないため本質的に吸収困難であり、言語の型システム、DDD の Aggregate、GRASP の Information Expert、いずれも判断基準を制約するが決定しない。

### 6.2 SPEC 駆動 + 中間ファイル化による消去

SPEC を唯一の保持対象とし、UC 以降を中間ファイルとして扱うことで、E 後半が解けなかった理由(早すぎる抽象化は誤った抽象を固定する、固定の解除コストが高い)が消滅する。

**消去の機序**:

- 抽象化は今世代の中間ファイルの構造に過ぎない
- 「誤った抽象」は SPEC 改訂 → 再生成で解消、累積コストなし
- 未来予測が不要なのは、未来の変化 = SPEC の変化 = 再生成、だから
- AI が「設計として良さそう」で導入した抽象も、暫定的に存在するだけで害がない

E 後半責務は **配置先を探す問題ではなく、発生抑止される責務** となる。ここでいう「再生成」が日常運用では局所再生成として実現されることは §8 で扱う。

### 6.3 SPEC 駆動が成立するための条件

- SPEC が再生成に必要なすべてを近似的に捕捉している
- 再生成コストが現実的(AI 以前は破綻していたため、AI 登場で初めて成立)
- 外部依存者は SPEC 由来のインターフェースに依存し、コード内部構造に依存しない
- legixy が SPEC→中間ファイルの依存をすべて追跡し、SPEC 起源を持たないコード片を孤児として検出する

条件 1 は「すべて」ではなく「近似的にすべて」とする。原理的に完全捕捉は循環するため(完全性を判定するには既に再生成結果を知っている必要がある)、運用上は目標漸近として扱う。

## §7 有向グラフの中央装置性

### 7.1 有向グラフが担う責務

SCP における有向グラフ(legixy graph.toml)は単なる追跡装置ではなく **中央装置** である。担う責務は以下に集約される。

- SPEC → 中間ファイル → SRC の依存記録(下降)
- 再生成時の整合性検証(SemanticChecker、SPEC 起源完全性)
- 影響範囲分析(下降方向の前方伝播)
- 修正時の変更箇所逆引き(上昇方向の探索)
- 質問票ループでの参照(前段ループでも修正フローでも)

### 7.2 修正フローの二経路

**経路 A: 明示経路** — ユーザーが SPEC のどこを変えるべきか知っている場合。「UC-103 のエラーハンドリングを変えたい」のように、SPEC レベルで変更箇所を指定できる。AI は下流伝播の追跡を支援する。

**経路 B: 探索経路** — ユーザーは結果(動作)で変更を表現するが、SPEC のどこを変えるべきか分からない場合。AI は有向グラフを逆引きして候補箇所を特定する。動作 → SRC → ... → DD → SEQ → RB → UC → SPEC の上流探索。

### 7.3 コンパイラ理論との対応

| コンパイラ | SCP |
|---|---|
| ソースマップ(行レベル) | 有向グラフのノード対応 |
| DWARF デバッグ情報 | 有向グラフのエッジ情報 |
| 実行時エラー → ソース行特定 | 動作要望 → SPEC ノード特定 |
| シンボルテーブル | SPEC 起源完全性検証 |

ただし SCP の有向グラフは **多対多の意味的依存グラフ** であり、コンパイラの 1 対 1 ソースマップを構造的に拡張したものである。

### 7.4 ライフサイクル全体管理

有向グラフはプロジェクトのライフサイクル全体で機能する。

| 期 | 中央装置の用途 |
|---|---|
| 開発期 | SPEC → SRC の降下追跡、整合性検証 |
| 受け入れ期 | 動作 → SPEC 逆引き、修正経路 |
| サポート期 | バグ報告 → 影響範囲分析、修正候補特定 |
| 拡張期 | 新規 SPEC の既存グラフへの統合 |
| ポーティング期 | バックエンド切り替え、SPEC 起源保持 |
| 終焉期 | 後継システムへの SPEC 移行、グラフ継承 |

すべて同じ有向グラフ操作で、違いは入力(何が変更要求か)と探索方向だけである。本章での有向グラフ記述は静的な側面に重点があり、動的な差分処理装置としての側面は §8 で扱う。

## §8 インクリメンタル・コンパイラとの対応

### 8.1 運用実態とコンパイラ・モードの収束

SCP の実運用では、SPEC を改訂するたびに全中間ファイルをフル再生成することはほとんど発生しない。日常運用では legixy の有向グラフを用いて変更の影響範囲を特定し、その範囲内の中間ファイルだけを局所再生成する。これは運用上の選択ではあるが、構造的に意味のある事実である。

すなわち SCP の実運用は **伝統的バッチコンパイラ(全ソースから全バイナリを生成する)ではなく、インクリメンタル・コンパイラ(依存グラフを保持し、変更影響範囲のみ再計算する)に構造的に対応する**。本章はこの対応を明示化する。

本書が §3 で示した「伝統的コンパイラとの対応」は構造原理の説明としては適切だが、運用実態の記述としては不十分である。日常運用は §3 が描く全段一括変換ではなく、§7 が描く有向グラフ駆動の局所更新である。この差分を埋めるのが本章の責務である。

### 8.2 二つのコンパイラ・モード

SCP は構造として二つのコンパイラ・モードを持つ。両者は同じ有向グラフ上の操作だが、操作の粒度と頻度が異なる。

**フル再生成モード**:

- 発動契機: ポーティング、言語切替、大規模リファクタリング、初回ビルド
- 頻度: プロジェクト生涯で稀
- 処理: SPEC 全体 → 全中間ファイル → 全 SRC を一括生成
- 冪等性: LLM の非決定性により、点としては保証不可。同値類(検証を通る集合)としては保証可能

**インクリメンタル・モード**:

- 発動契機: 日常運用、機能追加、バグ修正、SPEC 部分改訂
- 頻度: 日常的
- 処理: SPEC 差分 → 有向グラフで影響範囲確定 → 該当ノードのみ再生成
- 冪等性: 「同じグラフ状態 + 同じ変更操作 → 同じ次状態グラフ」として、legixy の決定論的処理で構造的に保証される

運用実態は主にインクリメンタル・モードで進行する。フル再生成モードは特殊状況でのみ発動する。

### 8.3 インクリメンタル・コンパイラ理論の参照

伝統的バッチコンパイラ(GCC、初期 Clang 等)とは別に、現代の処理系の多くはインクリメンタル・コンパイラとして設計されている。本書はこれらを参照領域として明示化する。

| 処理系 | インクリメンタル機構 |
|---|---|
| Rust Incremental Compilation | クエリ・ベースド評価、依存グラフ、メモ化キャッシュ |
| Salsa (Rust) | 一般化されたインクリメンタル計算フレームワーク、入力変更時の自動再計算 |
| TypeScript Compiler API | ファイル単位の差分処理、ProjectReferences、incremental build |
| Roslyn (C#) | セマンティック・モデルの増分更新、Workspace 抽象 |

**共通する原理**:

- 計算結果をメモ化する
- 入力の依存グラフを保持する
- 入力変更時、変更が影響するノードのみ再計算する
- 影響範囲はグラフ操作によって決定される
- 冪等性は「グラフ操作の決定性」として保証される

これらの原理は legixy + SCP の運用実態に直接対応する。

### 8.4 対応表

| インクリメンタル・コンパイラ | SCP | 備考 |
|---|---|---|
| 依存グラフ | legixy graph.toml | 差分処理の基盤 |
| クエリ・ベースド評価 | SPEC 変更 → 影響範囲計算 | 変更要求からノード集合を導出 |
| メモ化された中間結果 | 中間ファイル(UC/RB/SEQ/DD/TS/TC) | キャッシュとしての中間ファイル |
| 無効化(invalidation) | SemanticChecker drift detection | 変更により無効化されたノードの特定 |
| 増分再計算 | 局所再生成 | 無効化ノードのみ再生成 |
| 入力差分 | SPEC 差分 | 変更の起点 |
| 影響範囲計算 | 有向グラフの前方伝播 | 無効化の波及 |
| クエリ結果の安定性 | SPEC 起源完全性 | キャッシュの整合性保証 |

### 8.5 legixy の位置付けの精緻化

これまで legixy は本書で三つの側面から記述されてきた。三つは矛盾せず、参照する側面が異なる。

| 側面 | 記述章 | 参照領域 | 主たる性質 |
|---|---|---|---|
| 側面 1 | §3 | シンボルテーブル/DWARF | 静的情報の保持装置 |
| 側面 2 | §7 | ライフサイクル全体の中央装置 | 全期にわたる単一機構 |
| 側面 3 | §8 | インクリメンタル・コンパイラの依存グラフ | 動的な差分処理装置 |

本書の他章では側面 1、2 が主に使われていたが、**運用実態に最も近いのは側面 3** である。本書を読む際は、章ごとに参照されている側面が異なることを意識する必要がある。

### 8.6 冪等性の再定義

「コンパイラ構造」を主張する場合、冪等性は典型的に問われる。SCP は LLM を含むため伝統的バッチコンパイラと同じ意味での冪等性は持たない。これを「成立しない」と片付けるのではなく、冪等性を二種類に分離して再定義する。

**冪等性の二分**:

- **フル再生成の冪等性**: 「同じ SPEC → 同じ全中間ファイル」
  - LLM の非決定性により、点としては保証不可
  - 同値類(検証を通る集合)としては保証可能
  - フル再生成モードでの冪等性

- **グラフ操作の冪等性**: 「同じグラフ状態 + 同じ変更操作 → 同じ次状態グラフ」
  - legixy の決定論的処理として構造的に保証可能
  - 局所再生成の正当性根拠
  - インクリメンタル・モードでの冪等性

運用上問われる冪等性は後者である。インクリメンタル・モードでは「同じ SPEC 差分 + 同じグラフ状態 → 同じ影響範囲 + 同じ更新手順」が legixy の決定性によって保証される。**再生成内部の LLM 処理に非決定性があっても、再生成の発動位置と範囲は決定的に確定する**。

運用がインクリメンタル・モードに収束することで、フル再生成の非冪等性が日常運用に影響する経路が構造的に閉じる。これは「冪等性問題を運用で回避する」ではなく「運用実態が問うべき冪等性を別カテゴリに移している」と解釈すべきである。

### 8.7 二モード分離による既存章の再解釈

二モード分離は本書の既存章を二つに分類する視点を与える。

| 章 | 属するモード | 備考 |
|---|---|---|
| §4.5 マルチバックエンドの分岐点 | フル再生成モード | 言語切替時に発動 |
| §10.1 言語選択の事後変更可能性 | フル再生成モード | 稀に発動する可能性の主張 |
| §10.2 ポーティングコストの構造的削減 | フル再生成モード | 稀に発動する可能性の主張 |
| §7 有向グラフの中央装置性 | インクリメンタル・モード | 日常運用の主たる経路 |
| §10.3 ライフサイクル全体の単一機構による管理 | インクリメンタル・モード | 日常運用の主たる経路 |
| §10.4 知識の externalize の完了 | インクリメンタル・モード | 日常運用の主たる経路 |
| §2 前段ループ | 両モードに共通 | SPEC 改訂のたびに前段が走る |

この分類により、各主張がどのモードを前提にしているか、そして日常運用での発動頻度がどの程度かが明確になる。

## §9 自然言語コンパイラとしての SCP

### 9.1 ソース言語の性質

伝統的コンパイラのソース言語は形式言語(BNF で定義された構文)である。曖昧性は構文エラーとして検出され、人間が修正する。AI との対話的ループは存在しない。

SCP のソース(SPEC)は自然言語+構造(または半形式)で書かれる。曖昧性の検出に AI の理解能力が必要であり、検出結果を質問票として返す必要がある。

### 9.2 構造的差異

| 項目 | 伝統的コンパイラ | SCP |
|---|---|---|
| ソース言語 | 形式言語(BNF) | 自然言語+構造 |
| フロントエンドの曖昧性解消 | 構文エラーによる人間修正 | AI 質問駆動の対話ループ |
| パスの実行主体 | 決定論的プログラム | 役割割り当て LLM(非決定論) + 決定論的ツール(legixy) |
| IR の形式 | 形式記法(SSA 等) | 部分的に自然言語、部分的に形式記法 |
| 射程 | ソース → 実行可能バイナリ | ソース → 実行可能アプリ + ライフサイクル全体管理 |
| 再生成モデル | バッチ(再ビルド)+ インクリメンタル | 同じく二モード(§8) |

### 9.3 新カテゴリとしての位置付け

SCP は「**人間-AI 協調コンパイラ**」として、コンパイラ理論の延長線上にある新カテゴリとして位置付けられる。

既存コンパイラとの違いはソース言語の形式性とパスの実行主体の二点に集約される。それ以外の構造(パイプライン、IR、パス、依存グラフ、最適化、検証、インクリメンタル処理)はすべて既存コンパイラ理論で記述可能である。

**非決定論レイヤを決定論レイヤで縫う**: 本書全体に通底する設計原理は「非決定論的処理(LLM)を決定論的検証・追跡(legixy、SemanticChecker)で縫う」というアーキテクチャである。前段ループ(§2)、SemanticChecker(§7)、グラフ操作の冪等性(§8.6)はいずれもこの原理の系である。

この二層構造は伝統的コンパイラには存在しない。伝統的コンパイラは全層が決定論的だからである。SCP の新カテゴリ性の構造的根拠はここに集約される。

### 9.4 参照領域の変化

SCP を「人間-AI 協調コンパイラ」として位置付けることで、設計判断で参照すべき領域が明確になる。

- パスの差し替え可能性 → コンパイラの最適化パスの on/off 構造を参照
- IR の安定性 → コンパイラの IR 世代間互換性を参照
- パスの順序保証 → コンパイラのパス依存宣言を参照
- デバッグ情報の保持 → DWARF を参照
- 複数ターゲットへの同時出力 → LLVM のマルチターゲット構造を参照
- 差分ビルドの保証 → Salsa、Rust Incremental Compilation を参照
- キャッシュ無効化 → Roslyn、TypeScript Compiler API を参照

開発プロセス論ではなくコンパイラ理論を参照することで、検証可能性が高く、設計上の選択肢が体系化されている領域から知見を借用できる。

## §10 構造的含意

### 10.1 言語選択の事後変更可能性(フル再生成モード)

SPEC + UC が言語非依存に書ければ、言語選択は SPEC 完成後でも変更可能になる。SPEC を書いた時点では Rust 想定でも、後から Scala に切り替えても、SPEC は変えずに再コンパイルで対応できる。

これは現在のソフトウェア開発の常識を反転する。通常、言語選択はプロジェクト初期に決まり、後の変更は実質的に書き直しを要する。SPEC 駆動 + 中間ファイル化 + マルチバックエンドにより、言語選択が最適化の選択肢に降格する。

本主張はフル再生成モード(§8.2)に属する構造的可能性であり、日常運用での発動頻度は低い。

### 10.2 ポーティングコストの構造的削減(フル再生成モード)

「Java で書いたシステムを Go に移植する」のような作業は通常巨大な作業である。SPEC + UC が真のソースなら、移植はバックエンド切り替えだけになる。SPEC を変えず、生成ターゲットを変える。

これは EDA(Electronic Design Automation)で同じ Verilog 記述から ASIC 用と FPGA 用を生成するのと構造的に対応する。Verilog ソースは変えず、合成ターゲットを変える。

本主張もフル再生成モードに属する。日常運用ではなく、移行プロジェクトでの構造的可能性である。

### 10.3 ライフサイクル全体の単一機構による管理(インクリメンタル・モード)

有向グラフがすべての期で同一の中央装置として機能することで、現代のソフトウェア開発が達成していない統合が成立する。

| 期 | 現代の典型 | SCP |
|---|---|---|
| 開発 | Git + IDE + CI/CD | 有向グラフ + AI 駆動降下 |
| 受け入れ | テスト管理ツール | 有向グラフの逆引き |
| サポート | チケットシステム | 有向グラフ + 修正フロー |
| 運用 | 監視ツール | 有向グラフ + 動作ログ |
| ドキュメント | Confluence / Notion | SPEC + 中間ファイル(再生成可能) |

現代のツール群はそれぞれが整合性を人間の規律に依存する。SCP は有向グラフという単一の中央装置に集約することで整合性を構造的に保つ。本主張はインクリメンタル・モードに属し、日常運用の主たる経路を構成する。

### 10.4 知識の externalize の完了(インクリメンタル・モード)

通常、サポート期に必要な知識(設計時の意図、実装時の制約、過去の修正履歴)の多くは暗黙である。担当者の頭の中にあるか、散逸する。

SCP では SPEC + 有向グラフ + 質問票履歴のすべてが外部化されている。プロジェクトの知識が完全に成果物として保持される。

これにより担当者の入れ替わりが構造的に許容され、原設計者がいなくてもシステムが理解・修正可能になる。「20 年前のコードを書いた人がもういない、誰も読めない」という古典的レガシー問題が構造的に消える。

### 10.5 抽象パターンの組合せによる到達

SCP がコンパイラ構造に到達したことは、コンパイラ構造が「コンパイラ理論という特定領域の発明」ではなく、**複雑な変換問題に対する一般解として独立に到達可能な構造** であることを示唆する。

底辺の構造パターン(主語・責務・境界、責務配置、単一情報源、決定論/非決定論分離、検証段階の挿入)を保持し、それらを組み合わせる作業を続けると、結果としてコンパイラ的構造に到達する。本書の場合は ICONIX、TDD、ロバストネス図、SPEC 駆動などのソフトウェア工学の蓄積を経由しており、純粋な独立到達ではないが、コンパイラ理論を直接参照したわけでもない。**経路の独立性と到達点の同型性は別の事柄である**。

## §11 限界と境界

### 11.1 形式性と意図性の区別

SCP は形式的正しさ(SPEC 完備、降下整合)を機械検証可能にする。しかし意図的正しさ(SPEC が真の要求を反映しているか、生成アプリが要求を満たすか)は機械検証不可能であり、開発者が動かして確かめる必要がある。

これは伝統的コンパイラと同じ構造的限界である。コンパイラはどんなに高度になっても「コンパイル成功」しか保証せず、「意図通りに動く」かは別軸である。

加えて、形式性ゲートそのものも AI の検出能力に依存する近似であり、「検出されなかった不足」の存在を排除しない。これは伝統的コンパイラには存在しない第三の軸である。

### 11.2 SPEC 駆動の前提条件への依存

SPEC 駆動が成立しない環境では、SCP の多くの利点が機能しない。具体的には:

- SPEC が再生成に必要なすべてを近似的に捕捉できていない場合 → E 後半が本流に残る
- 再生成コストが現実的でない場合 → 中間ファイル化が破綻する
- 外部依存者がコード内部構造に依存している場合 → 再生成が外部に副作用を与える
- legixy の SPEC 起源完全性が破れている場合 → 孤児コードが累積する

### 11.3 認知特性への依存

SCP の設計と運用は空間認識能力に基づく抽象的構造物のカタログ保持を前提とする側面がある。設計者がロバストネス図的分解(主語・責務・境界・前提)を自然に扱えない場合、SCP の中段(分析層、設計層)の運用が困難になる可能性がある。

これは認知特性の個人差に基づく制約であり、SCP の構造的欠陥ではない。ただしすべての開発者が同じように運用できるわけではないことを認識する必要がある。

### 11.4 二モード分離が前提する運用

§8 で定義した二モード分離は、運用がインクリメンタル・モードに収束することを前提する。以下の条件が崩れると、この前提も崩れる:

- legixy の有向グラフが SPEC 改訂に追随できる粒度を維持していること
- 影響範囲計算が現実的時間で終わる規模であること
- 局所再生成の結果がフル再生成の結果と整合的であること(局所と全体の整合性)
- 複数開発者が同時に SPEC を改訂したときの意味レベルコンフリクトが解決可能であること

これらは現時点で構造的に成立する見込みがあるが、実証は限定的である。

### 11.5 適用範囲の明示

**本書が主張すること**:

- SCP は前段ループを加えると伝統的コンパイラ構造と対応関係を持つ
- SCP の実運用はインクリメンタル・コンパイラ構造と対応関係を持つ
- これらの到達は構造的な事実であり、設計時の意図ではなく事後認識である
- 対応関係により、コンパイラ理論の知見を SCP 設計に転用可能
- マルチバックエンド、ポーティングコスト削減、ライフサイクル全体管理が構造的に成立する
- 冪等性は二種類に分離され、運用上問われる冪等性は legixy の決定性で保証される

**本書が主張しないこと**:

- SCP はすべての開発プロジェクトに適している
- SCP は伝統的開発手法より優れている
- SCP は経験のない開発者でも運用可能
- SCP はコンパイラ理論の研究成果である
- SCP は伝統的コンパイラと数学的同型である
- 本書の主張のすべてが実証されている

## §12 一文での総括

> SCP は、SPEC を唯一のソースとし、UC 以降を中間ファイルとして扱う直列パイプラインに前段ループを加えることで、伝統的コンパイラと構造的に対応する人間-AI 協調コンパイラに至り、その実運用は legixy 有向グラフ駆動の局所再生成として、インクリメンタル・コンパイラと構造的に対応するモードに収束する。対応関係は意図された設計ではなく、底辺の構造パターンの組合せから事後的に到達したものであり、これによりコンパイラ理論(バッチおよびインクリメンタル両方)の知見をソフトウェア開発プロセス全体に転用する経路が開かれる。
