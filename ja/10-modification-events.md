# 10 — 修正イベントフロー

このドキュメントは、**イベント駆動の修正フロー** を規定する。新規作成フロー (chain SPEC → UC → ... → SRC) だけでなく、既存連鎖への変更を **イベント** として整理し、統一フローで処理する。

## 1. なぜイベント駆動か

**新規作成中心** の構造では、修正は補足的にしか扱われない。しかし実際の開発では新規作成より修正の方が頻度が高く、これは設計上の盲点になりやすい。本章はこの盲点を埋める。

イベントで分類することの利点:

1. **役割と分離**: 「誰が」検出したか (DEV / Support) ではなく「何が」起きたかで分類するので、Support 層実装の有無に依存しない
2. **共通フロー**: 4 つのイベントは共通中核操作 (`investigate` → `impact` → 修正候補生成) を共有し、コードと文書の重複を避ける
3. **将来拡張**: V-DRS Support 層が実装される将来、Support 層は単に「これらのイベントを外部からトリガーする経路」を提供すれば良い。フロー本体は本章のままで動く

## 2. 4 つのイベント

| イベント | 起点 | trace 方向 | 主な対象 | 例 |
|---|---|---|---|---|
| **新規作成** (`creation`) | プロジェクト初期 / 独立した新規連鎖 | なし | 新規連鎖全体 | プロジェクト立ち上げ時の最初の SPEC |
| **仕様追加** (`spec-add`) | 既存システム内の新 SPEC | 既存への接続点を `upstream` 探索 | 新 SPEC + 既存との接続点 | 既存 APP に新機能 SPEC を追加 |
| **仕様変更** (`spec-change`) | 既存 SPEC / UC | `downstream` (影響範囲特定) | 変更ノードの下流全体 | 既存 SPEC §3.2 の責任分担を変更 |
| **不具合修正** (`defect-fix`) | 既存 SRC (or 任意の既存ノード) | `upstream` (原因分析) + `downstream` (影響) | 起点周辺 + 上流原因 | unit test green 後の機能不全発見 |

新規作成と仕様追加の境界は実用上「**既存 RTM に接続するかしないか**」で判定する。既存 RTM と接続しないケースはほぼ起こらないため、実質的には仕様追加が新規作成を包含する。

## 3. 共通中核操作

4 イベントのすべてが共有する操作シーケンス。本章ではこれを **共通中核** と呼ぶ:

```
[起点ノードの特定]
  ├─ 起点が確定的: そのまま次へ
  └─ 起点が不確定: embedding 類似度 + キーワードで候補列挙
                    確信度閾値以上 → 自動確定
                    閾値未満 / 拮抗 → 候補リストを開発者に提示

[trace 探索]
  ├─ 上流方向 (--upstream): 原因分析が必要なイベント (主に defect-fix)
  └─ 下流方向 (--downstream / impact): 影響範囲特定 (主に spec-change)

[成果物参照]
  探索範囲内の各ノードの内容を AI が読み込む

[修正候補生成]
  ├─ 変更対象ノードの一覧
  ├─ 各ノードの修正案 (diff 形式)
  ├─ 影響を受けるテストの一覧
  └─ 根拠 (依拠した上流成果物への参照)
```

この共通中核は V-DRS-ARCH-001 v1.1.0 追補 §6.5 と同一構造である。本章は V-DRS §6 の **origin-agnostic 統一版** として位置付けられる (DEV / Support の起点区別を排除している)。

## 4. 各イベントの詳細フロー

### 4.1 不具合修正 (`defect-fix`)

最も頻発するイベントであり、本フレームワークが最初に最適化すべきイベント。書き手が議論した「**SRC 検出 → 上流追跡 → ドキュメント不備検査 → 修正 → インクリメンタル再構築**」の流れを具体化する。

**起点**: 不具合の症状 (テスト失敗、機能不全、意図しない動作)

**フロー**:

```
1. 不具合の検出
   (誰がいつどこで見つけたかは記録するが、フロー分岐の要因にしない)

2. 不具合箇所の SRC 特定
   2a. 確定的な場合 (例: スタックトレース付きのエラー)
       → そのまま次へ
   2b. 不確定な場合 (例: 「何かおかしい気がする」)
       → legixy の embedding 検索で SRC 候補列挙
       → 候補リストを開発者に提示、選択を待つ

3. investigate で関連ドキュメント収集
   SRC → TC → TS → DD → SEQ → RB → UC → SPEC の連鎖を辿る

4. 上流ドキュメントの不備検査 (Reviewer subagent)
   収集された範囲全体に対して 9 観点 (review-guidelines/perspectives.md) で
   不備を検査する。特に効くのは以下の観点:
     - [Consistency]: 三者整合性違反が不具合原因か
     - [Coverage]: 未テスト観点が不具合原因か
     - [Doc]: DD と SRC の乖離が不具合原因か
     - [Layer]: レイヤ汚染が不具合原因か
   検出された不備を VERDICT 形式で出力

5. 原因の判定
   検出された不備のうち、本当に当該不具合の原因と判断できるものを開発者が選別
   (false positive を排除、複数原因の可能性を保持)

6. ドキュメント修正
   特定された原因ドキュメントを修正する
   原則として SRC を直接 patch するのではなく、上流ドキュメントを直す
   (理由: 不具合の真の原因は多くの場合上流にある。SRC patch は対症療法)

7. impact で再構築範囲の特定
   修正されたドキュメントから下流に向けて影響範囲を計算

8. インクリメンタル再構築
   §5 (本章) のインクリメンタル再構築フロー参照

9. legixy check
   全体整合性の確認

10. [Recurrence] 観点による再発防止
    review-guidelines/perspectives.md の [Recurrence] 観点を発火
    以下のいずれかを選択:
      - trace-check に grep ベース検出を追加
      - perspectives.md に新観点を昇格
      - ai-antipattern.md に新カテゴリを追加
      - ADR で例外として記録
      - 何もしない (理由を commit message に明記)
```

#### 4.1.1 起動方法

```bash
# 開発者が明示的に起動
/defect-fix <SRC-ID-or-symptom-description>

# 例:
/defect-fix SRC-APP-042
/defect-fix "保存ボタンを押しても history に記録されない"
```

#### 4.1.2 ハードルール 6 との関係

「仕様書とテストコードは実装着手後に変更しない」(ハードルール 6) は、**気まぐれな変更を禁止する** 規律であって、**不具合の根本原因として上流を修正する** ことは想定された動作である。本フロー Step 6 でドキュメントを修正することはハードルール 6 違反ではない。

ただし、Step 5 で原因判定が曖昧なまま Step 6 に進むと、対症療法的な SPEC 改変が発生する。これはハードルール 6 が警戒する pattern である。

### 4.2 仕様変更 (`spec-change`)

既存の SPEC / UC を意図的に変更するイベント。仕様変更要求の受領、設計改善の発意などが起点。

**フロー**:

```
1. 変更対象ノードの確定 (常に確定的、開発者が指定)

2. impact で下流影響範囲の特定
   変更ノードから下流の全成果物を列挙

3. 影響範囲のレビュー (Reviewer subagent)
   変更内容が下流の各成果物にどう波及するかを 9 観点で評価

4. 人間承認 (ハードルール 1: SPEC 変更は人間承認必須)
   変更内容と影響範囲を確認、承認する

5. 変更の適用
   対象 SPEC / UC を修正

6. インクリメンタル再構築
   §5 のフロー参照

7. legixy check
   全体整合性の確認

8. ゲート再評価
   影響を受けたゲート (08-gates.md) を再実行
```

#### 4.2.1 起動方法

```bash
/spec-change <SPEC-ID-or-UC-ID>
```

### 4.3 仕様追加 (`spec-add`)

既存システムに新規 SPEC を追加するイベント。新規ニーズの発見、機能拡張要求などが起点。

**フロー**:

```
1. 新 SPEC の起票
   templates/SPEC-template.md からコピーして起草

2. 前段ループ (03a-frontend-pass.md)
   Raw SPEC → QSET → SPP → FCR → Accepted SPEC

3. 既存 SPEC との衝突検査 (Reviewer subagent)
   新 SPEC が既存 SPEC と矛盾しないか [Consistency] 観点で検査
   既存ドメイン用語との一致を [Layer] 観点で検査

4. 接続点の特定
   新 SPEC が既存連鎖のどの UC / RB と接続するかを embedding で推定

5. 人間承認 (ハードルール 1)

6. 新 SPEC の確定
   FCR.frontend_status = ACCEPTED

7. 新規連鎖の構築
   通常の chain SPEC → UC → ... → SRC を起動
```

#### 4.3.1 起動方法

```bash
/spec-add <description-or-file-path>
```

### 4.4 新規作成 (`creation`)

プロジェクト初期化時の、既存 RTM と接続しない新規連鎖の構築。実用上は仕様追加と同じフローを使えるが、Step 3 (既存との衝突検査) と Step 4 (接続点の特定) はスキップする。

通常、プロジェクト立ち上げ時にのみ発生する。

## 5. インクリメンタル再構築

`defect-fix` Step 8 および `spec-change` Step 6 で発火する、変更されたドキュメントから下流を選択的に再生成する操作。`09-compiler-lens.md` の **Salsa-style 差分計算** を literal に運用する。

### 5.1 基本原則

```
[修正されたドキュメント]
        ↓
[impact で下流ノードを列挙]
        ↓
[各下流ノードを順番に再評価]
        ├─ 修正前の上流に依存していた部分を identify
        ├─ 修正後の上流に基づき、当該部分を再生成
        └─ 修正されていない部分には触らない
        ↓
[各再生成ノードに対し AI Reviewer を発火]
        ↓
[VERDICT が REQUEST_CHANGES なら再生成内容を修正]
        ↓
[VERDICT が NEEDS_HUMAN なら開発者承認を待つ]
        ↓
[全ノードが APPROVE になったら legixy check 実行]
```

### 5.2 「変更された」の判定

何が「変わった」と見なすかは legixy v0.4.0-alpha4 の機能群を活用する:

- **テキスト diff**: 単純な行レベル差分。明示的な変更箇所の特定
- **subnode 粒度**: ドキュメント全体ではなく、変更された見出し配下だけ再生成 (`refresh-subnodes`)
- **embedding drift**: 意味的に等価な変更 (誤字修正、句読点など) を識別、再生成不要と判定

### 5.3 再構築の人間承認ゲート

すべての再生成を AI に任せると、ハードルール 1 (SPEC 変更は人間承認必須) と矛盾する。以下の境界を設ける:

| 再生成対象 | 承認 |
|---|---|
| UC / RB / SEQ (構造翻訳層) | AI Reviewer の APPROVE で足りる |
| DD | API surface の変更を含む場合は人間必須、それ以外は AI Approve 可 |
| TS / TC | AI Reviewer の APPROVE で足りる |
| SRC | AI Reviewer の APPROVE で足りる |
| SPEC / UC の責任分担変更 | 人間必須 (`spec-change` イベントの一部) |

### 5.4 非決定論性への対策

AI は同じ入力でも異なる出力を返し得る (LLM の non-determinism)。完全に決定論的なインクリメンタル再構築は原理的に不可能である。これに対する対策:

1. **構造の再構築は冪等**: どのノードを再生成すべきか、依存関係はどうか、は決定論的に計算できる (legixy の graph 操作)
2. **内容の再生成は非冪等**: 個々のドキュメント・コードの内容は毎回違うものが出る可能性がある。これは drift detection の閾値で「意味的に等価」を判定して許容する
3. **人間承認ゲート**: 非冪等性が問題になる境界 (SPEC 変更、API 凍結) では人間判断が必須

詳細は `09-compiler-lens.md` の「冪等性の二層化」セクションを参照。

## 6. 再帰検出 (3 サイクル超過警告)

修正イベントの結果として `legixy check` が新たな不整合を検出すると、新しい修正イベントが発火する。これは V-DRS-ARCH-001 v1.1 §8-6 の「修正フローの再帰」と同じ構造である。

### 6.1 サイクルの記録

各修正イベント発火時に、`.scp/event-log.txt` に記録する:

```
2026-05-27T13:00:00Z | defect-fix | SRC-APP-042 | parent: -      | cycle: 1
2026-05-27T13:15:00Z | spec-change | SPEC-APP-005 | parent: cycle-1 | cycle: 2
2026-05-27T13:32:00Z | defect-fix | SRC-APP-051 | parent: cycle-2 | cycle: 3
```

### 6.2 3 サイクル超過時の挙動

```bash
scripts/count-fix-cycle.sh  # サイクル数をチェック
```

サイクル数が 3 を超えると、警告が出力される:

> ⚠ 修正フローが 3 サイクルを超過しました。これは個別修正で対処すべき問題ではなく、設計レベルの見直しが必要なシグナルです。
>
> 推奨アクション:
>   1. 現在のサイクルを停止する
>   2. これまで修正されたノード群を点検する
>   3. ADR を起票して設計判断の根本的な見直しを記録する
>   4. 必要なら SPEC レベルでの再設計を検討する

3 サイクル超過後も継続することは可能だが、その場合は ADR にスキップ理由を記録すること。

### 6.3 サイクル中断の判定

開発者がサイクル中断を判断する基準:

- 修正が「別の修正を誘発するだけ」になっている (収束しない)
- 修正のたびに新しい不整合が増えている (発散している)
- 修正対象が初期起点から離れた成果物に広がっている (impact が予想を超えて拡大している)

## 7. AI レビュアによる不備検査 ([AI-Antipattern] 観点の拡張)

`defect-fix` Step 4「上流ドキュメントの不備検査」で AI Reviewer が陥る罠を `review-guidelines/ai-antipattern.md` §G として追加する。詳細は同ファイルを参照。

主要な罠:

- **G-1 浅い読解**: 上流ドキュメントを章節レベルでしか確認せず、subnode レベルの不備を見逃す
- **G-2 過剰修正**: ドキュメントの軽微な記述揺れを「不備」として検出し、本来の不具合と混同する
- **G-3 修正の依存無視**: ドキュメント A を修正したが、A に依存する B/C を再生成しないままにする
- **G-4 対症療法**: SRC レベルでの patch を「修正」とし、上流の真の原因を放置する
- **G-5 サイクル放置**: 3 サイクル超過の警告を受けても、個別修正を継続する

## 8. イベントログ (`.scp/event-log.txt`)

修正イベントの発火履歴を append-only で記録する。フォーマット:

```
ISO8601 timestamp | event-type | trigger-node-id | parent-cycle | cycle-number | result-verdict | archive-path
```

例:

```
2026-05-27T13:00:00Z | defect-fix  | SRC-APP-042   | -        | 1 | APPROVE  | .scp/event-output/2026-05-27T13-00-00.md
2026-05-27T13:15:00Z | spec-change | SPEC-APP-005  | cycle-1  | 2 | APPROVE  | .scp/event-output/2026-05-27T13-15-00.md
2026-05-27T13:32:00Z | defect-fix  | SRC-APP-051   | cycle-2  | 3 | NEEDS_HUMAN | .scp/event-output/2026-05-27T13-32-00.md
```

このログは:

- 再帰サイクル数の追跡 (Step 6 の根拠データ)
- 修正フローの統計分析 (どのイベントが最も頻発するか)
- 監査記録 (V-DRS Support 層が将来導入されたとき、Support 起点のイベントもここに記録される)

に使われる。

## 9. V-DRS Support 層への自然な拡張

本章のフローは、書き手の現在の context (solo dev、Support 層未実装) で実証される。Support 層が将来実装された場合、Support 層は以下を実装するだけで本章のフローが再利用される:

1. **外部からのイベント発火経路**: ユーザからの不具合報告を `defect-fix` イベントとして発火する仕組み
2. **FixProposal の保留状態**: Support 起点の `defect-fix` は即座に Step 1 から実行されるのではなく、「発火候補」として一時保留され、開発者の確認を経てから本フローに入る
3. **read-only クエリイベント**: 操作問合せ (V-DRS-ARCH-001 §5.3) のような RTM 読み取りのみのイベント (将来の拡張)

FixProposal は新しい type code ではなく、`GAP` の状態の一つとして表現できる:

```
GAP-APP-NNN
  status: pending-event-confirmation  # 新規 status (発火候補)
  event-type: defect-fix
  trigger-source: external-support-report
  related-id: SRC-APP-042
```

開発者が確認して承認すると、status が `active` に変わり、本章のフローが起動する。

## 10. 段階的導入

イベントフローは基本フロー(新規作成チェーン)の完全運用の上に追加する。段階的に有効化する:

| Phase | 内容 | 検証目的 |
|---|---|---|
| **Phase 0** | 基本フロー(本章なし)を運用 | 9 観点 AI Reviewer の基本動作確認 |
| **Phase E1** | `defect-fix` イベントのみ実装 | 最も頻発するイベントから着手、フローの妥当性検証 |
| **Phase E2** | `spec-change` イベントを追加 | 下流影響分析の検証 |
| **Phase E3** | `spec-add` イベントを追加 | 既存 RTM との衝突検査の検証 |
| **Phase E4** | インクリメンタル再構築を有効化 | drift detection と subnode 粒度の本格運用 |
| **Phase E5** | 再帰検出・3 サイクル警告を有効化 | 修正フローのメタ運用 |

各 Phase で `ai-antipattern.md` §G に新パターンが追加されることが想定される。Phase E1 で発生しなかった罠が、Phase E4 (再構築) で初めて現れる可能性が高い。

## 11. 検証されていないこと

本章は思考実験を経て構築された仕様であり、運用検証されていない。特に以下は **やってみないと分からない**:

- AI Reviewer が `defect-fix` Step 4 で上流ドキュメント群を本当に深く読めるか (G-1 浅い読解への耐性)
- 3 サイクル警告が現実の頻度で発動するか (常に発動なら閾値が低すぎ、全く発動しないなら検出ロジックが甘い)
- インクリメンタル再構築の drift threshold が現実の修正で機能するか (常に全部再生成 or 何も再生成しない、の両極端に倒れないか)
- イベントログ (`.scp/event-log.txt`) のフォーマットが分析に耐えるか (修正履歴の grep でパターン抽出が可能か)
- subnode 粒度の再構築が `refresh-subnodes` 機能で実用に耐えるか
- 開発者が頻繁に `/defect-fix` を起動する operational rhythm が成立するか (操作負荷の問題)

これらの未検証性を Phase E1 から段階的に明らかにしていく。各 Phase の検証結果は `ai-antipattern.md` §G、`perspectives.md` `[Recurrence]`、または ADR として記録する。
