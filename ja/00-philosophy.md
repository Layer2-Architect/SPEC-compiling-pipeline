# 00 — このプロセスの哲学

## 1. 第一目的: 品質偏向防止

> **確率論的変換器を構造材として採用するソフトウェア開発プロセスは、品質最大化ではなく品質偏向防止を第一目的とすべきである。**

これが本プロセスの最上位の設計原理である。以降のすべての設計判断はこの原理から導かれる。

### 1.1 確率論的変換器とは何か

伝統的コンパイラ(GCC, Clang)は確定論的変換器である。入力 x に対して関数 f によって出力 y = f(x) を返す。同じ入力からは常に同じ出力が出る。

AI(LLM)は確率論的変換器である。入力 x に対して確率分布 P(y|x) を返し、そこから 1 つの y をサンプリングする。同じ入力でも、サンプリングのたびに出力が異なる。temperature を 0 にしても、attention の数値精度ゆらぎ・batch 化・tokenizer の曖昧性により、完全な決定論にはならない。

本プロセスは AI を SPEC, UC からの下流変換(SRC 生成までの全工程)を担うコンパイラとして使う。すなわち f: x → y ではなく f: x → P(·|x) を構造材として採用する。

### 1.2 確率論的変換器に「品質最大化」を要求することの構造的問題

品質最大化は、確定論的変換器では明確に定義可能である(実行速度を上げる、コードサイズを縮める、最適化を強化する)。

確率論的変換器では、品質最大化は分布上の複数の量のいずれかを意味する:

- 期待値 E[quality(y)] の最大化
- 最頻出値 mode[quality(y)] の最大化
- 下限 min[quality(y)] の最大化(=最悪値の上限を上げる、minimax)

伝統的開発プロセス論は暗黙のうちに前 2 者を採用する。「ベストプラクティスを集めて最適化する」のは期待値か最頻出値の最大化である。

しかし確率論的変換器をコンパイラとして使うとき、構造的に正しいのは 3 番目の minimax である。理由は 3 つある:

**理由 1: コンパイラの位置付けは下限保証である**

伝統的コンパイラに対する暗黙の要求は「生成されたバイナリは少なくとも仕様通りに動く」という下限保証である。「平均的にはうまく動く」ではコンパイラとして機能しない。AI を SPEC → SRC のコンパイラとして使うなら、同じ下限保証が要求される。「どんなサンプリングでも、最悪ケースが catastrophic failure に至らない」という構造的保証が要る。

**理由 2: 確率分布の左裾は実質的に観測されない**

実装で使われるのは分布全体ではなく、分布からの単一サンプル y である。同じ SPEC を 1000 回サンプリングして分布を観察することは現実的にしない。100 クレート規模で UC が 50〜200 本あれば、その数だけサンプリングが発生し、確率的に必ず左裾サンプル(分布の悪い領域からのサンプル)が混入する。

品質を「平均値」で評価していると、実装で実際に使われる単一サンプルが左裾から出たときに無防備になる。防御の構造的に正しい方法は、左裾サンプルそのものを上げる(AI のチューニング、原理的に限界あり)か、左裾サンプルが出ても catastrophic failure に至らない経路を別に用意する(本プロセスが採用する方向)のどちらかである。

**理由 3: 確率分布の偏向は一方向に蓄積する**

LLM の確率分布 P(y|x) は均等ではなく、訓練データの分布によって偏向している。同じ SPEC に対して「もっともらしい解」が複数あっても、LLM は特定の解(訓練データで頻出する解)を高確率で選ぶ。

100 クレート規模で UC ごとに 1 サンプルずつ生成すると、全体として LLM の訓練分布バイアスに偏った設計が累積する。個別 UC では問題に見えないが、システム全体では一方向に歪む。これは確定論的コンパイラには存在しない、確率論的変換器固有の性質である。

### 1.3 品質偏向防止の定式化

以上から、確率論的変換器をコンパイラとして使うときに要求すべきは:

**要求 1**: 確率分布 P(y|x) の左裾サンプルが catastrophic failure に至らないこと
**要求 2**: 確率分布 P(y|x) のバイアスが特定方向に蓄積しないこと

これが「品質偏向防止」の正確な定式化である。

「品質最大化」が右裾(良いサンプル)を上げることだとすれば、「品質偏向防止」は左裾の catastrophic failure を防ぐことと、分布全体の bias 累積を防ぐことである。

AI の内部チューニング(プロンプト、fine-tuning、RAG)は分布を P(y|x) から P'(y|x) に変える試みだが、根本的に確率論的であることは変わらない。一方、左裾の catastrophic failure を防ぐ努力と bias 累積を防ぐ努力は、AI の外側の構造(冗長系、多方法論、独立検証チャネル、人間観察)で実装可能で、限界がない。

**本プロセスは AI の内部チューニングではなく、AI の外側の構造設計で品質偏向防止を実装する。**

### 1.4 既存分野との連結

確率論的変換器または不確実性のあるシステムに対する設計原則は、既存分野で確立されている:

- **Robust statistics**: median が mean より使われるのは、データ分布の左裾(outlier)に対する頑健性を中心傾向の正確性より優先するため。これは minimax の発想である。
- **Adversarial robustness(機械学習)**: 訓練分布外の入力に対するモデルの頑健性を、平均精度より別の指標として扱う。
- **Safety engineering**: FMEA / FTA で失敗経路を体系的に列挙し、各経路に対する防御を多重化する。航空機の冗長系、原子炉の多重防護、医療機器の interlock 機構はすべて「最大性能ではなく最悪ケースの上限」を設計目標とする。

本プロセスは、これらの分野で確立された「確率論的または不確実性のあるシステムに対する設計原則」を、AI コンパイラという新しい構造材に適用するものである。独立の発見ではなく、既存の robust design 思想の software engineering 領域への移植として位置付けられる。

ソフトウェア工学が伝統的に決定論的システムを前提にしてきたため、robust design の語彙と方法論が software engineering 領域では確立されていない。本プロセスはこの欠落を埋める一つの解である。

## 2. 品質偏向防止のための構造的方策

第一目的(§1)を達成するための方策は 5 つに分類できる。これらは独立した防御線として機能し、各々が別の偏向源に対する対抗手段を提供する。

### 2.1 入力分布の狭隘化 — 前段ループ

確率論的変換器の出力分布 P(y|x) は入力 x に依存する。入力が曖昧であれば分布は広がり、左裾サンプルが出やすくなる。**入力の不完備を解消することで分布を狭め、左裾サンプル発生確率を構造的に下げる**。

これが前段ループ(`03a-frontend-pass.md`)の本質的役割である。Raw SPEC を QSET ⇄ SPP ⇄ FCR の反復で正規化することは、入力 x の品質を上げて P(y|x) の分散を狭める操作に対応する。

ハードルール 9(SPEC は前段ループで FCR.frontend_status = ACCEPTED に到達していなければ TP[SPEC] / UC 着手禁止)は、入力分布が十分狭くなる前に下流のサンプリングを開始することを禁止する構造的拘束である。

### 2.2 出力偏向の検出可能化 — 観察可能化と TP/GAP

確率論的変換器の出力 y の偏向は、単一サンプルでは検出不能である。1 サンプル見ると「これは妥当な解だ」と判定できるが、偏向は分布全体の性質であり、個別サンプルからは読み取れない。

**観点を生成して仕様にぶつけることで、偏向を「観察可能な欠落」に変換する**。これが TP / GAP ループ(`03-spec-level-tdd.md`)の本質的役割である。

「観点」は「テストケース」ではなく、**仕様文書に問いかける質問のリスト**である。観点ナレッジベース(`docs/perspectives/`)を AI の外側に持つことで、AI が単独では思いつかない観点を強制的に注入する。これは AI の確率分布の偏向に対して、外部から網羅性を要求する経路である。

ハードルール 2(GAP がクローズしないうちに次フェーズへ進まない)は、検出された偏向を放置したまま下流に進むことを禁止する。

### 2.3 多重独立検証経路 — 方法論並置

各方法論には固有の盲点(その方法論が想定しない品質次元での失敗を捕捉できない性質)がある。**複数の方法論を意図的に並置し、各方法論の盲点を別の方法論が構造的にカバーすることで、特定方向への偏向蓄積を防ぐ**。

本プロセスが並置する方法論:

| 方法論 | 担当する品質次元 | 出典 |
|---|---|---|
| TDD(コードレベル) | 実装の検証可能性 | Beck/Cunningham |
| 仕様レベル TDD(TP/GAP) | 仕様の網羅性 | 本プロセス |
| ICONIX 二段化(RBA/SEQA/RBD/SEQD/DD) | 構造整合性 | Jacobson OOSE 1992 + Rosenberg & Stephens 2007 |
| 三者整合性検証(両流派) | 動的整合 + 静的整合 | Jacobson 流 + ICONIX 流 |
| traceability(legixy) | 仕様⇄実装の追跡可能性 | コンパイラ理論(シンボルテーブル/DWARF) |
| Property-based testing | 不変条件の網羅 | proptest / hypothesis 系 |
| Model checking | 並行性・プロトコル | loom / Spin |
| AT(人間観察) | 暗黙知・ドメイン慣行 | 受け入れテスト系 |
| NFR | 性能・並行性・リソース | 非機能要件系 |
| ADR | 設計判断の根拠 | Michael Nygard |
| AI Antipattern カタログ | AI 特有の盲点 | 本プロセス |
| 9 観点 AI レビュア層 | 機械検証と人間判断の中間 | 本プロセス |
| 配送軸契約適合(CTR/DLV/TC[DLV]) | 機能の外部公開＝凍結境界契約への適合 | 本プロセス(v1.0, `12-delivery-layer.md`) |

これらは「ベストプラクティスの寄せ集め」ではなく、各方法論の盲点を別の方法論が構造的にカバーする配置になっている:

- TDD の盲点(設計が偏る) → ICONIX 構造翻訳が上流で構造を確定
- ICONIX の盲点(性能・並行性が後回し) → NFR と loom が独立検証
- 形式的検証の盲点(仕様自体の妥当性) → AT が人間観察で検出
- AT の盲点(頻度が低い) → TP/GAP が観点ベースで仕様段階に持ち上げる
- AI Reviewer の盲点(同質モデル) → 機械検証層と AT が独立経路として存在
- ADR の盲点(記録のみ) → AI Reviewer の `[Doc]` がドキュメント追従性を検証
- **機能チェーン GREEN の盲点(契約サーフェス＝CLI/API/MCP の不適合・孤児が機能 GREEN では露見しない)
  → 配送軸の TC[DLV] 実バイナリ E2E + 構造の WARN-escalate ゲートが独立経路として強制検証(v1.0, `12-delivery-layer.md`)**

ハードルール 10(ICONIX 二段化レイヤ汚染禁止 + 三者整合性検証必須)は、多重独立検証経路の構造を保つための拘束である。

### 2.4 確率分布外側からの観察 — AT

AI Reviewer は AI の確率分布の内側にいる(同じか相関の高い分布で動作する)。AI が見落とすパターンを別の AI が見落とす構造的限界がある。これは prompt 工夫や異モデル投入で部分的に緩和できるが、根本的には解消しない。

**AI の確率分布の外側からの観察として、AT(人間観察)が構造的に必要である**。`07-at-and-nfr.md` に定める AT は、実機操作で AI が記述レベルで捕捉できない現実との乖離を検出する。

本プロセスでは AT が「下流品質の最終防衛線」として位置付けられる。AI レビュア層がどれだけ精緻化されても、AT は省略不可能である。

AT で発見された問題は GAP[UC] / GAP[SPEC] として記録され、新観点として `perspectives.md` に昇格する(§2.5)。

### 2.5 偏向パターンの事後蓄積 — ai-antipattern.md と Recurrence

確率論的変換器の偏向は完全には予測不能である。「どのような偏向が発生するか」は運用するまで分からない。

**発生した偏向を事後に記録し、次回検出可能性を上げる経路として、`ai-antipattern.md` カタログと `[Recurrence]` 観点が機能する**。

`ai-antipattern.md` は AI 特有の罠の構造的分類で、新パターンが発見されたら追記される(human-on-the-loop)。`[Recurrence]` 観点はバグ修正 PR ごとに「次回どう防ぐか」(trace-check 追加 / perspectives 昇格 / ガイドライン追加 / ADR 例外 / 何もしない)の明示を要求する。

これは AI の確率分布を直接変えるのではなく、AI の出力に対する事後フィルタを育てる方向である。「**改善しようがないが、観察可能で学習可能**」な構造として偏向を扱う。

## 3. 人間関与境界の構造的根拠 — SPEC, UC のみ

本プロセスにおける人間関与は SPEC と UC に限定される。RBA 以降(SEQA, RBD, SEQD, DD, TS, TC, SRC)は AI 自律実行 + AI Reviewer 層 + AT の組み合わせで品質保証する。

これはハードルール 11 として規範化される(`README.md` のハードルール一覧参照)。

### 3.1 関与境界の構造的根拠

§1 で示した品質偏向防止の論理は、人間関与の位置付けにも適用される。

**最も重要な防御線に人間関与を集中する**ことが、「介入を減らす」よりも構造的に正しい設計である。理由は 3 つある:

**根拠 1: 入力分布の品質が全防御線の前提**

§2.1 で示した通り、確率論的変換器の出力分布は入力に依存する。入力(SPEC, UC)が高品質であれば、下流のすべての防御線の効率が上がる。逆に入力が曖昧であれば、下流でいくら多重防御しても catastrophic failure 発生確率は構造的に下がらない。

人間の限られた認知資源を入力品質向上に集中することで、防御線全体の効率が最大化される。

**根拠 2: 確率分布の外側からの観察として SPEC + UC が機能する**

§2.4 で AT が「分布外側からの観察」として位置付けられたが、SPEC と UC も同じ位置付けである。SPEC と UC は AI が確率的に生成する成果物ではなく、人間がドメイン知識と判断で確定する成果物である。

これらは確率論的変換器の入力であると同時に、確率論的変換器の出力に対する基準でもある。下流で AI が偏向した設計を生成しても、それが SPEC と UC に整合しているかを照合することで偏向が検出される。SPEC と UC が AI の確率分布の内側にあると、この照合経路が機能しなくなる。

**根拠 3: catastrophic failure の根本原因の大半は上流にある**

`10-modification-events.md` の defect-fix フローが示す通り、不具合の真の原因は多くの場合上流のドキュメント不備にある。SRC レベルの patch は対症療法であり、上流を直さなければ同種の問題が再発する。

下流(RBA 以降)に人間関与を分散させると、上流の不備を下流で「埋め合わせ」する力学が働きやすくなる(`ai-antipattern.md` A-1, A-2 参照)。人間関与を SPEC + UC に集中することで、不備を発見した時点で必ず上流に戻る規律が構造的に保たれる。

### 3.2 関与境界の操作的定義

人間が直接関与する作業:

- SPEC 起草・改訂・最終承認
- QSET への回答
- SPP の承認 / 却下
- UC のフロー妥当性レビュー・最終承認
- TP[SPEC] / TP[UC] の観点漏れ確認
- GAP の重要度判断・close 判定
- AT 実施(被験者観察)
- NFR 閾値設定
- ADR 承認
- リリース判断
- 修正イベント(`/defect-fix`, `/spec-change`, `/spec-add`)の発火と承認
- 品質基準変更(`review-guidelines/`, `.trace-engine.toml`, ハードルール本文)の最終承認

人間が直接関与しない作業:

- RBA, SEQA, RBD, SEQD, DD の生成
- TS, TC, SRC の生成
- 上記下流成果物の品質レビュー(AI Reviewer 層が代行)
- レイヤ汚染検査(機械検証で代行)
- 三者整合性検証の実施(AI Adversary が代行、判定結果のみ AT 段階で照合)

例外: ハードルール 7(境界 API は DD で凍結)の凍結判断は人間必須として残るが、これは「凍結する API のリスト」を AI が提示し、人間が一括承認する形で運用する(API ごとに DD 全文を読まない)。

### 3.3 関与境界が成立する前提条件

人間関与を SPEC + UC に絞ることが成立する前提:

- SPEC と UC が下流の AI 自律実行に十分な情報密度を持つこと(`03-spec-level-tdd.md` の TP[UC] 観点で担保)
- AI Reviewer 層が機能していること(`08-gates.md` の VERDICT 機構、`review-guidelines/` の 9 観点)
- AT が独立検証チャネルとして実施されていること(`07-at-and-nfr.md`)
- `ai-antipattern.md` カタログが運用で育成されていること(human-on-the-loop)
- legixy による traceability が機能していること(`06-trace-engine.md`)

これらの前提が崩れている状態で関与境界を SPEC + UC に絞ると、下流品質が無防備になる。Phase E0(検証フェーズ)で前提条件の成立を経験的に確認する手順は `guides/adoption-phases.md` を参照。

## 4. 既存哲学概念の再位置付け

SCP までの哲学記述(信頼境界の上流移動、観察可能化、形式性ゲート、埋め合わせ禁止、ICONIX 採用、traceability 採用)は本プロセスの中核として有効である。これらは §1 の品質偏向防止という上位原理の下に、それぞれ独立した防御線として位置付け直される。

| 既存概念 | 品質偏向防止の枠組みでの位置付け |
|---|---|
| 信頼境界の上流移動 | 確率論的変換器の入力 x の品質向上による分布 P(y|x) の狭隘化(§2.1)|
| 観察可能化 | 確率論的変換器の出力 y の偏向を観点ぶつけで検出可能化(§2.2)|
| 形式性ゲート | 入力分布が狭くなる前のサンプリング開始を禁止する構造的拘束(§2.1、ハードルール 9)|
| 埋め合わせ禁止 | 入力分布の不備を下流サンプリングで補うことの禁止(§2.1 + §3.1 根拠 3)|
| ICONIX 採用 | 多重独立検証経路における構造整合性チャネル(§2.3)|
| ICONIX 二段化 | 抽象側・具体側を別チャネルとして並置することによる偏向検出(§2.3)|
| traceability 採用 | 偏向検出経路の追跡可能性確保(§2.3 の検証経路の中央装置、`09-compiler-lens.md` §7)|
| AT の独立性 | 確率分布外側からの観察(§2.4)|
| AI Antipattern カタログ | 偏向パターンの事後蓄積(§2.5)|

これらは独立して語ることもできるが、品質偏向防止という統一原理の下に整理することで、各概念が「なぜ必要か」が一貫した論理で説明可能になる。

---

## 5. なぜ「品質最大化」が誤った目標か

「品質最大化」を目標にした開発プロセスは確定論的システム(コードとコンパイラと OS)の文脈で発達してきた。決定論的システムでは「品質を上げる」と「品質の偏りを防ぐ」は事実上同じ問題である。最大値を上げれば最小値も上がる。

確率論的システムでは、この同一性が崩れる。最大値を上げても最小値が下がる場合がある(分布が広がるとき)。最大値を上げなくても最小値だけを上げる方法がある(分布を狭めるとき)。

ソフトウェア工学のベストプラクティス論は、暗黙のうちに「分布」という概念を扱わずに作られてきた。確率論的変換器が開発プロセスに組み込まれるのは生成 AI 時代に初めて起きていることで、そこで要求される品質論はソフトウェア工学の伝統的枠組みから組み替える必要がある。

「ベストプラクティスを集めて最適化する」発想で本プロセスを評価すると、19 タイプコード・5 層 ICONIX・複数の独立検証チャネル・前段ループは「重い」「冗長」「儀式的」に見える。

しかし品質偏向防止の枠組みでは、これらは多重防御の必要層数として位置付けられる。航空機の冗長系・原子炉の多重防護・金融システムの defense-in-depth と同じ構造である。各層が「単独では非効率」だからこそ、「合わせると catastrophic failure に強い」。

100 クレート規模で 5 年以上運用するプロダクトでは、効率と冗長性のトレードオフは冗長性側に倒すのが正しい。短期 prototype なら逆だが、本プロセスは明確に長期運用向けである(§7 参照)。

「重い」批判への構造的応答: それらは多重防御として必要な層数である。減らすなら、どの偏向に対する防御を犠牲にするかを意識的に選ぶ必要がある。意識せずに削減すると、削減された層が担当していた偏向が無防備になる。

## 6. 検出できる領域とできない領域

| 領域 | 検出手段 | 担当する §2 防御線 |
|---|---|---|
| 仕様の網羅性(境界値・エラー・権限・状態遷移・異常系) | TP / GAP | §2.2 |
| 仕様↔実装の整合性 | TDD + legixy | §2.3 |
| 性能・並行性・リソース管理 | NFR + ベンチマーク + loom | §2.3 |
| 暗黙知・ドメイン慣行・前提不一致 | AT | §2.4 |
| AI 特有の盲点パターン | ai-antipattern.md カタログ | §2.5 |
| 確率論的変換器の左裾サンプル | 上記すべての多重防御 | §2.1〜2.5 |

「AT で検出される領域」は原理的に仕様レベル TDD で検出不能である。`perspectives.md` 昇格機構を通じて、AT で発見された観点が次回の TP 生成で検出可能化する。これにより仕様レベル TDD で検出できる範囲が漸進的に拡大する(§2.5)。

## 7. このプロセスが向いている領域・向かない領域

### 向いている

- 公開 API がある(FFI 境界、ライブラリ、SDK、サービス API)
- 仕様の正しさが品質に直結する(決済、医療、シミュレーション、コンテンツ生成パイプライン)
- 複数の実装系(言語・ランタイム)が境界 API を共有する
- AI 生成を多用するプロジェクト
- 長期運用(5 年以上想定)
- 大規模(50 UC 以上、複数クレート規模)
- catastrophic failure のコストが高い(規制対象、安全クリティカル、可用性要求)

### 向かない

- 探索フェーズの prototype(仕様自体が探索対象)
- 短期 throw-away スクリプト
- 既存システムへの単発バグ修正
- 小規模(数 UC、単一ファイル規模)
- catastrophic failure のコストが低い(個人用ツール、内部 hack)

探索フェーズと本プロセスの境界は「**境界 API を凍結すべき段階に入ったか**」で判断する。凍結フェーズに入ったら本プロセス、それ以前は通常の TDD で良い。

長期運用 + 大規模 + catastrophic failure コストが高い、という 3 条件は重なって本プロセスの適用妥当性を決める。1 つでも欠ければ、多重防御のコストが利益を上回る可能性が高い。

## 8. 段階的導入の前提

新規プロジェクトでも全層を一気に立ち上げる必要はない。Phase 構成は `guides/adoption-phases.md` を参照。

各 Phase の効果指標は **「TS 以降で発見される仕様起因バグの件数」** であるが、これは平均的バグ数の減少を測定しているのではなく、**確率論的変換器が左裾サンプルを出したときに、それが上流に逆流して仕様レベルで検出される頻度**を測定している。バグが減ることそのものより、バグの根本原因が仕様層まで遡れること、それが多重防御の機能の証拠である。
