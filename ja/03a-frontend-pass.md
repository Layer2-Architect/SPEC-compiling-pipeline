# 03a — 前段ループ（フロントエンド・パス）

このドキュメントは Raw SPEC を受け取って Accepted SPEC まで正規化する **前段ループ** の作業時に参照する。SPEC レベル TDD ループ（`03-spec-level-tdd.md`）に進む **前** の必須フェーズである。

## 1. 前段ループの目的

人間が初稿として書いた SPEC は必ず不完備である。前段ループは、その不完備を質問票駆動の対話ループで解消し、後段（TP/UC/RB/SEQ/DD/TS/TC/SRC）の機械的展開が成立する状態まで SPEC を **正規化** する。

> **核心**: 前段ループは AI に設計判断を委任するものではない。AI は開発者の入力をもとに、後段変換に必要な情報が揃っているかを確認する **フロントエンド検査器** として振る舞う。意図性のゲートではなく、形式性のゲートである。

## 2. 全体フロー

```
Raw SPEC（人間が書いた初稿）
  ↓
AI による必要項目テンプレート仮充填（AI 内部、人間に見せない）
  ↓
不足・曖昧・矛盾・境界不明・例外未定義の検出（AI 内部）
  ↓
QSET 発行（質問票、人間が読む）
  ↓
開発者の回答
  ↓
SPP 発行（SPEC 差分案、人間が読む）
  ↓
開発者の承認 ──[却下]──→ QSET 修正に戻る
  ↓ [承認]
SPEC 更新（差分反映）
  ↓
フロントエンド検証 → FCR 発行
  ↓
FCR.frontend_status = NEEDS_QUESTIONNAIRE ──→ 新規 QSET 発行に戻る
FCR.frontend_status = ACCEPTED               ──→ 03-spec-level-tdd.md（TP[SPEC] 着手）へ
```

**反復回数の上限は設けない。** 質問票が出続けるのは、SPEC がまだ後段変換可能な状態に達していないことを意味する。打ち切るなら ADR で「なぜ前段ループを途中で確定するか」を記録する（ハードルール 9 への明示的スキップ扱い）。

## 3. 必要項目テンプレート（AI 内部構造）

AI が SPEC の充足度を判定するための内部検査表。**人間が手で埋めるフォームではない**。

| 項目群 | 内容 | 未充足時 |
|---|---|---|
| 目的 | システムが何を達成するためか | 目的質問 |
| 利用者・外部主体 | ユーザー種別、外部システム、管理者、ゲスト | 主体質問 |
| 主要機能 | 観測可能な機能、ユースケース候補 | 機能質問 |
| 入力 | ユーザー入力、外部 API 入力、ファイル、イベント | 入力質問 |
| 出力 | 画面表示、保存結果、通知、API 応答、生成物 | 出力質問 |
| 状態 | ライフサイクル、状態遷移、保存状態、一時状態 | 状態質問 |
| 例外・失敗時挙動 | 異常系、外部依存失敗、権限不足、タイムアウト、重複、欠損 | 例外質問 |
| 制約 | 技術制約、運用制約、法令、セキュリティ、データ保持 | 制約質問 |
| 非機能要件 | 性能、可用性、保守性、監査性、可観測性、移植性 | 非機能質問 |
| 受け入れ条件 | どの状態になれば要求を満たしたと判定するか | 完了条件質問 |
| 禁止事項 | してはいけない挙動、許可しない状態、避ける実装方針 | 禁止事項質問 |

AI は SPEC から各項目を抽出して仮充填し、未充足項目を QSET の質問に変換する。テンプレート自体は AI 内部で完結し、QSET / SPP / FCR のいずれにも直接コピーされない。

## 4. QSET 生成規則

AI は内部検出をそのまま人間へ提示しない。開発者が回答可能な自然言語の質問票（QSET）へ変換する。

| 内部検出 | 質問票の形式 | 例 |
|---|---|---|
| 未定義語 | 用語確認質問 | 「ここでいう『管理者』はシステム管理者ですか、組織の管理者ですか?」 |
| 複数解釈 | 選択肢付き質問 | 「削除は論理削除ですか、物理削除ですか?」 |
| 例外未定義 | 失敗時挙動質問 | 「外部 API が失敗した場合、再試行/エラー表示/処理保留のどれですか?」 |
| 境界不明 | 責務範囲質問 | 「通知文面の生成は本システムの責務ですか、外部サービスの責務ですか?」 |
| 矛盾 | 衝突解消質問 | 「SPEC-12 では即時反映、SPEC-31 では承認後反映とあります。どちらを採用しますか?」 |
| 非機能不足 | 運用条件質問 | 「同時利用者数、応答時間、ログ保存期間について希望値はありますか?」 |

### QSET 生成の禁止事項

- AI が勝手に仕様を確定してはならない（ハードルール 1）
- 複数の設計方針を、根拠なく一つに決めてはならない
- 人間に内部テンプレートの全項目を埋めさせてはならない
- 質問は **後段コンパイルに必要な不足・曖昧性・矛盾に結びついていなければならない**

## 5. SPP 生成規則

開発者が QSET に回答したら、AI はその回答を反映した SPEC 差分案（SPP）を発行する。

- SPP は QSET の各質問に対応した形で差分を提示する
- 差分は SPEC の修正前/修正後を明示する diff 形式（unified diff 推奨）
- 設計判断を含む差分には根拠と代替案を併記する
- SPP は人間が承認するまで SPEC に反映してはならない（ハードルール 1）

## 6. フロントエンド検証（FCR 発行）

開発者が SPP を承認 → SPEC 更新 → AI がフロントエンド検証を実行 → FCR を発行する。

| 検証項目 | 内容 | 失敗時 |
|---|---|---|
| 必須項目充足 | 後段コンパイルに必要な項目が埋まっているか | 質問票へ戻す |
| 用語一貫性 | 同一語が複数意味で使われていないか | 用語確認質問 |
| 主体一貫性 | 同じ主体の責務が矛盾していないか | 責務確認質問 |
| 状態遷移充足 | 主要状態に遷移元、遷移先、異常系が存在するか | 状態質問 |
| 例外経路充足 | 失敗時、権限不足、外部依存失敗が未定義でないか | 例外質問 |
| 境界整合性 | システム内外、外部依存、人間作業の境界が不明でないか | 境界質問 |
| 矛盾不在 | 要求同士が競合していないか | 衝突解消質問 |
| UC 生成可能性 | SPEC から UC 候補を生成できる程度に閉じているか | 不足質問 |

### FCR の判定式

```
if required_template_complete
   and glossary_consistent
   and no_blocking_ambiguity
   and no_blocking_contradiction
   and exception_paths_sufficient
   and boundary_sufficient
   and usecase_generation_possible
   and human_approved:
       frontend_status = ACCEPTED
else:
       frontend_status = NEEDS_QUESTIONNAIRE
```

**FCR の `frontend_status` が ACCEPTED でない SPEC は TP[SPEC] / UC 着手禁止**（ハードルール 9）。

## 7. 反復と打ち切り

```
SPEC 改訂 → QSET → 開発者回答 → SPP → 承認 → 検証 → FCR
                                              │
                                              ├─ ACCEPTED → 完了
                                              └─ NEEDS_QUESTIONNAIRE → 次の反復へ
```

- 各反復で QSET / SPP / FCR を **新規 ID** で発行する（古いものを書き換えない、履歴を残す）
- 同一 SPEC に対する FCR は ID 連番が最大のものを「現在の状態」とみなす
- 反復 N 回目の QSET の本文 metadata に `**反復回数**: N` を記載する

## 8. 前段ループが保証することと保証しないこと

### 保証すること

- 後段コンパイルに必須な情報の未充足を検出し、QSET として返す
- 検出可能な矛盾、曖昧性、境界不明、例外未定義を減らす
- 開発者が承認した SPP だけを SPEC へ反映する
- TP[SPEC] / UC 着手できる程度に SPEC を閉じる

### 保証しないこと

- 開発者の真の意図を AI が完全に理解していること
- 承認された SPEC が市場・利用者・組織にとって正しいこと
- 生成されるアプリが人間の期待を完全に満たすこと
- 将来の要求変更が発生しないこと
- AI が検出すべき全ての不足・曖昧性・矛盾を実際に検出すること

前段ループは **形式性のゲートであり、意図性のゲートではない**。意図性の判定は SRC 生成後の人間確認（AT、`07-at-and-nfr.md`）に位置する。形式性ゲートそのものも AI の検出能力に依存する近似であり、完全性ではなく到達度として扱う。

## 10. ハードルール 9 と前段ループの関係

```
ハードルール 9: SPEC は前段ループで FCR.frontend_status = ACCEPTED 状態を経ていなければ TP[SPEC] / UC 着手禁止
```

このルールは `scripts/trace-check.sh` が grep ベースで機械検証する。具体的には:

- 各 SPEC ファイルに対して、`docs/frontend-pass/check-results/FCR-<AREA>-*.md` のうち `**対象 SPEC**: <SPEC-ID>` を持つものを探す
- 該当 FCR の中で **ID 連番が最大のもの** の `**frontend_status**:` を読む
- ACCEPTED であれば pass、NEEDS_QUESTIONNAIRE または FCR 未発行なら fail

FAIL 状態の SPEC を親に持つ TP / UC が存在すれば trace-check は exit != 0 で失敗する。

## 11. 前段ループのスキップ

緊急対応・既存プロジェクトへの段階的導入等で前段ループをスキップする場合、以下の手順を踏む:

1. `docs/adr/ADR-<AREA>-NNN_frontend-pass-skip-<SPEC-ID>.md` を発行
2. SPEC ファイルに `**前段スキップ**: ADR-<AREA>-NNN` を metadata として記載
3. `scripts/trace-check.sh` の grep 検査はこの metadata を見て pass 扱いに切り替える

スキップは ADR で記録された場合のみ pass。記録なきスキップは検査 fail のまま。

スキップが常態化すると前段ループの効果が失われる。ADR の累積件数を早期警戒指標として monitoring する。

## 12. SPEC 改訂時の前段ループ再走

既存 SPEC を改訂する場合、改訂後の SPEC に対して **前段ループを再走する** ことが原則。

- 軽微な誤字訂正 → スキップ可（ADR 不要、コミットメッセージで十分）
- 用語追加、例外経路追加、責務範囲変更 → 再走必須
- 機能追加、SPEC 構造変更 → 再走必須

判断基準: 「この変更で後段（UC 以降）に新しい未充足が発生する可能性があるか」。Yes なら再走。
