# 暗黙的矛盾検出パターンライブラリ

このドキュメントは SPEC・UC・RBA 等の上流成果物に潜む **暗黙的矛盾** を検出するための、構造化されたパターンライブラリである。

> **位置付け**: 本ライブラリは **観点ナレッジベース** の一部であり、規範ではない。テストベッドでの観察を経て効果が実証されてから、規範文書に統合するかを判断する。現時点では Adversary 役 LLM の検査補助として使用する参考資料。

## 1. 目的

明示的矛盾(SPEC 内で直接競合する記述)は LLM が比較的検出しやすい。しかし **暗黙的矛盾**(遠距離の論理依存関係の破綻、責務原則の逸脱、状態遷移の不完備など)は単純な照合では捕捉困難。

本ライブラリは、暗黙的矛盾が出現しやすい **構造的パターン** を列挙し、Adversary 役 LLM および人間レビュアーが探索すべき箇所を構造化する。

### 1.1 補完関係

| 検査層 | 検出対象 | 検出手段 |
|---|---|---|
| 前段ループ FCR | 必須項目欠落、明示的矛盾、用語不一致 | テンプレート照合、grep ベース |
| TP/GAP | 観点不足、網羅性違反 | 観点ナレッジベース照合 |
| **本ライブラリ** | **暗黙的矛盾、構造的逸脱** | **パターン照合 + Adversary 検査** |
| AT | ドメイン慣行違反、暗黙の前提違反 | 人間判断、実装後検証 |

本ライブラリは前段ループと AT の間に位置し、構造的検出が可能な暗黙的矛盾を補完的に拾う層。

## 2. 検出パターンの構造

各パターンは以下の構造で記述する。

```
パターン名: 自然言語の短い識別子
トリガー: 検出すべき箇所を特定するための起点
周辺要素: トリガーの周囲で確認すべき情報
検出規則: 矛盾の判定基準
修正方針: 検出された場合の対応指針
具体例: 検出パターンの例示
```

この構造は **「書かれていない事象を周辺の構造から検出する」** という方法論的アイデアを技術仕様に適用したもの。

## 3. パターン一覧

現時点で 5 つのパターンを定義する。テストベッドで観察された新規パターンは順次追加する(育てる資産)。

### 3.1 責務原則の逸脱

| 項目 | 内容 |
|---|---|
| **パターン名** | 責務原則の逸脱 |
| **トリガー** | RBA/RBD で Boundary/Control/Entity の役割と矛盾する操作・データ保持を発見 |
| **周辺要素** | 該当主語の役割定義、操作内容、関連する他主語の責務 |
| **検出規則** | Boundary が永続データを保持している / Entity がビジネスロジックを実行している / Control が永続データを保持している |
| **修正方針** | 責務を適切な主語に移動、または新規主語を追加 |

**具体例**:
- RBA で「ログイン画面」(Boundary) が「セッション状態」を保持している → Entity「セッション」に移動すべき
- RBA で「注文」(Entity) が「在庫引当ロジック」を実行している → Control「在庫引当処理」に移動すべき
- RBA で「注文確定処理」(Control) が「注文履歴」を保持している → Entity「注文履歴」に移動すべき

### 3.2 状態遷移の空白

| 項目 | 内容 |
|---|---|
| **パターン名** | 状態遷移の空白 |
| **トリガー** | SPEC で Entity の状態 A と状態 B が言及されているが、A → B の遷移経路が明示されない |
| **周辺要素** | Entity の全状態定義、状態を変える操作、状態遷移を駆動する Control |
| **検出規則** | 状態リスト内に遷移経路が定義されていない状態ペアが存在する |
| **修正方針** | 遷移経路を明示する Control 操作を SPEC に追加、または遷移不可能なら明示 |

**具体例**:
- 注文状態に「ドラフト」「確定済」「キャンセル済」が定義されているが、「ドラフト → キャンセル済」の経路がない
- ユーザー状態に「未認証」「認証済」「ロック中」があるが、「ロック中 → 未認証」の遷移が定義されない
- 各 Entity の状態定義を全パターン展開して未定義遷移を洗い出す

### 3.3 概念の非回収

| 項目 | 内容 |
|---|---|
| **パターン名** | 概念の非回収 |
| **トリガー** | SPEC §X で導入された概念が、後続セクションや関連 UC/RBA で参照されないまま消える |
| **周辺要素** | 概念の定義箇所、概念を使う想定箇所(操作、状態、関連) |
| **検出規則** | SPEC で定義された全概念について、参照回数が定義回数 1 のみ |
| **修正方針** | 概念を実際に使う、または概念定義を削除(不要な概念) |

**具体例**:
- SPEC §2 用語表に「優先顧客」が定義されているが、後続のどの UC でも「優先顧客」が使われない
- SPEC §4 で「ロックダウンモード」が概念として導入されているが、どの状態遷移にも現れない
- 機械検証可能: 用語表の各エントリに対して grep で参照頻度を測定

### 3.4 責任の非命名

| 項目 | 内容 |
|---|---|
| **パターン名** | 責任の非命名 |
| **トリガー** | SPEC/UC で「対応する」「処理する」「行う」のような **責任主体を伴わない動詞句** を発見 |
| **周辺要素** | 同一文脈内のアクター・主語・主体 |
| **検出規則** | 動詞句に対して責任主体が明示されない箇所が存在する |
| **修正方針** | 責任主体を明示、または主体を新規 SPEC エントリとして追加 |

**具体例**:
- SPEC §6.3 に「外部 API 失敗時はエラーログを記録し、再試行する」とあるが、「誰がそれをするか」が不明
- SPEC §7 「通知が送信される」(受動態) — 誰が通知主体か
- 受動態と無主語動詞句を grep で抽出し、責任主体の有無を確認

### 3.5 要件の非解消

| 項目 | 内容 |
|---|---|
| **パターン名** | 要件の非解消 |
| **トリガー** | SPEC 内に直接的に競合する要件が共存するが、どちらを優先するかが明示されない |
| **周辺要素** | 競合する両要件の文脈、関連する不変条件、エラー方針 |
| **検出規則** | 同一概念に対して矛盾する要件が共存し、優先規則が定義されない |
| **修正方針** | 優先規則を明示、または要件を統合する設計判断を ADR として記録 |

**具体例**:
- SPEC §3 「注文確定は即時に在庫を引き当てる」と SPEC §7 「在庫操作は夜間バッチで反映する」が共存
- SPEC §5 「データは永続化される」と SPEC §9 「処理途中のデータは揮発する」の境界が不明
- これは前段ループの「矛盾不在」検査と重なるが、本パターンは **遠距離の競合** に焦点

## 4. 適用手順

### 4.1 SPEC 確定後の検査

前段ループ FCR が ACCEPTED に到達した SPEC に対して、本ライブラリを適用する。

```
手順:
  Step 1 — SPEC を対象に各パターンを照合する Adversary プロンプトを起動
  Step 2 — 各パターンについて検出候補を列挙
  Step 3 — 候補リストを人間に提示
  Step 4 — 人間が候補を判定:
              真の暗黙的矛盾 → SPEC 修正 → 前段ループ再走
              誤検出         → 候補を却下、Adversary プロンプト改善
              判定保留       → AT で検証する項目として記録
```

### 4.2 抽象層 GREEN 確定前の検査

ICONIX 抽象層(RBA/SEQA)の GREEN 確定前にも本ライブラリを適用する(`04-iconix-layer.md` §11 の補強)。

```
追加検査項目:
  - RBA に対して 3.1 責務原則の逸脱、3.4 責任の非命名を照合
  - SEQA に対して 3.2 状態遷移の空白、3.4 責任の非命名を照合
  - SPEC ⇄ RBA 関係に対して 3.3 概念の非回収、3.5 要件の非解消 を照合
```

### 4.3 機械検証可能な部分

以下は grep ベースで部分的に機械検証可能:

| パターン | 機械検証可能な部分 | 機械検証不可能な部分 |
|---|---|---|
| 3.1 責務原則の逸脱 | ステレオタイプと操作内容の単純照合 | 概念領域の意味的妥当性 |
| 3.2 状態遷移の空白 | 状態リストからの全ペア展開と未定義遷移検出 | 「遷移不可能」が妥当な判断か |
| 3.3 概念の非回収 | 用語表エントリの参照頻度カウント | 概念定義の質的妥当性 |
| 3.4 責任の非命名 | 受動態・無主語動詞句の抽出 | 主語省略が文脈で許容されるか |
| 3.5 要件の非解消 | 競合キーワード(「即時」と「バッチ」等)の共起 | 競合が真の矛盾か単なる別側面か |

機械検証可能な部分は `scripts/trace-check.sh` の拡張として実装する価値がある(将来の選択肢として保留)。

## 5. 縦断パターン(将来拡張)

同一プロジェクト・同一作成者の SPEC 群から、固有の「いつもの SPEC スタイル」を抽出し、新規 SPEC で逸脱を検出する手法。

```
手順:
  Step 1 — プロジェクトの過去 SPEC 群から以下を抽出:
              - 用語使用パターン
              - 責任主体の命名規則
              - 状態定義の粒度
              - エラー方針の構造
              - 例外経路の網羅度
  Step 2 — 新規 SPEC でこれらのパターンと逸脱を検出
  Step 3 — 逸脱を「意図的な拡張」か「無意識の不整合」かを人間が判定
```

これは「同じプロジェクトでは似た書き方が一貫しているはず」という前提に立つ。

**現時点では未実装、テストベッドで効果検証してから判断する**。

## 6. 限界

本ライブラリの構造的限界を明示する。

| 限界 | 内容 |
|---|---|
| **網羅性なし** | 列挙されたパターンは「これまでに観察された典型」であり、新パターンは観察ベースで追加される。完全リストではない |
| **誤検出のリスク** | パターン照合は構造的トリガーに依存するため、文脈で正当化される箇所も検出してしまう可能性 |
| **意味的判定は AI 依存** | 各パターンの真の判定は Adversary 役 LLM の能力に依存。LLM 能力が低いと誤判定する |
| **ドメイン依存性** | 一部パターンはドメイン慣行に依存。一般的な業務系・対話系 SPEC を想定 |
| **形式論理の限界** | 暗黙的矛盾の真の検出は形式論理的に決定不能な部分を含む。本ライブラリは近似的検出 |

これらの限界を理解した上で、AT(受け入れテスト)による最終検証と組み合わせて運用する。

## 7. 参考

### 7.1 方法論的着想元

本ライブラリの構造化手法(トリガーと周辺要素から暗黙の構造を検出する設計)は、Void Structure Specification(VoidStructureSpec)から **方法論的アイデアのみ** を借用した。

具体的に借用したのは:
- 検出パターンを「トリガー + 周辺要素 + 検出規則」の三要素で構造化する手法
- 暗黙的なものを周辺の明示的構造から推定する設計原理

Void Structure Specification の概念体系(L2-D / L2-I、Void 構造、認知的署名)そのものは本ライブラリには含まれない。本ライブラリは技術 SPEC 領域に独立に再構成されたものであり、Void Structure Specification への依存はない。
