# 13 — 完成品適合検査（PAI）

このドキュメントは **PAI（完成品適合検査 / Product Acceptance Inspection）** の作業時に参照する。
PAI は AT / NFR / VAL と並ぶ**独立検証成果物**であり、**ビルド済みの完成品**（実行ファイル・公開 API・RPC・
MCP サーバ・ファイル形式など）を、**作者と独立した主体が・黒箱で・実環境において**、凍結境界契約（CTR）と SPEC に
適合しているか検査する。

> ⚠️ **本章追加の背景（経験的根拠）**: legixy の完成品で「全機能チェーン GREEN ＋ `trace-check` PASS ＋ 配送層
> TC[DLV] pass」にもかかわらず、**契約違反 54 件**が残存していた（`defect-root-cause` 参照）。それらを捕捉したのは
> 唯一、作者と独立した外部の黒箱適合検査だった。**機能 GREEN は契約適合を意味しない**。この穴は本プロセスの
> 第一目的（品質偏向防止＝多重独立検証経路、`00-philosophy.md` §2.3）に対する構造的欠落であり、PAI はそれを埋める。

---

## 1. なぜ独立した完成品検査が必須か

本プロセスは既に複数の検証層を持つが、**「完成品を、作者と独立に、黒箱で、契約観点から検査する経路」だけが
欠けていた**。

- **TC[GREEN]（機能軸）/ TC[DLV]（配送軸, `12-delivery-layer.md`）** は、いずれも **Author が同一チェーン内で
  白箱的に**書く。確率論的変換器（AI）の盲点は、同一作者が書くテストに**そのまま継承**される（`00-philosophy.md`
  §2.4「AI Reviewer は分布の内側にいる」と同型の構造）。TC[DLV] は**既知の契約項目の回帰防止**には有効だが、
  **作者が想定しなかった契約違反の発見**には構造的に弱い。
- **AT（`07-at-and-nfr.md`）** は分布の外側からの独立観察だが、射程が **UX・暗黙知・前提不一致**に限定され、
  「機能的正しさ・契約適合は TC で担保済み」を**前提**にしている。legixy はこの前提が崩れることを実証した。
- 結果、**完成品の契約適合**を独立に検査する主体が存在せず、「GREEN による品質過信」（`00-philosophy.md` §1）が
  蓄積した。

PAI は、AT が UX に対して果たす「分布外側からの独立観察」を、**契約・機能の完成品適合**に対して果たす。

### 1.1 PAI の二つの目的（本質）

PAI の存在意義は、単に「完成品のバグを見つける」ことではない。**より本質的には、SCP プロセス自体の穴を
見つけうる**ことにある。

1. **完成品の不適合の発見**（表層の目的）。
2. **SCP プロセス自体の構造的欠落の発見**（本質的な目的）。
   PAI が捕捉した不適合が「**全機能チェーン GREEN ＋ trace-check PASS ＋ TC[DLV] pass をすり抜けて完成品に残った**」
   ものであるなら、それは個別バグであると同時に、**そのクラスの不適合を上流で止められなかったプロセスの穴**の兆候
   である。legixy の 54 件はこの典型で、`defect-root-cause` の根本原因（RC-1: 凍結契約の適合テストが成果物として
   不在 / RC-2: CLI 統合層がチェーン孤児 …）と改善提案（P-1〜P-6）を導いた。これらの穴を塞いだのが本章（PAI）と
   `12-delivery-layer.md`（配送層）である。

> したがって PAI 失敗は二段で戻す。**症状の修正**（/defect-fix・/spec-change）に加え、**「なぜ全チェーンを通過したか」
> のプロセス根本原因分析**を行い、必要なら SCP 自体を改善する（§5、`10-modification-events.md` の [Recurrence]）。
> PAI は、プロセスが自分自身の盲点を発見するための、**プロセス外側からのフィードバック源**である。

---

## 2. PAI の定義

- **独立タイプ（chain 外）**。typecode = `PAI`。ID = `PAI-<AREA>-NNN`。場所 `docs/product-acceptance/`。
- **対象**: リリース候補の**ビルド済み完成品**（ソース木の単体テストではない）。CLI バイナリ・公開 API・RPC・
  MCP サーバ・生成ファイル形式など、**凍結境界契約（CTR）が公開を規定するサーフェス**。
- **環境**: **実環境**（実モデル・実フィクスチャ・実 OS・実依存）。hermetic 化しない。完成品が実際に動く条件で検査する。
- **判定**: 各ケースを `PASS` / `要改善`（指摘 ID 付き）/ `未実施`（理由付き）で記録する。`要改善` が 1 件でもあれば
  リリースゲート不通過。

PAI 本体（検査計画＋結果報告）が成果物であり、検査を駆動するハーネス（スクリプト・フィクスチャ）は補助物として
PAI から参照する。

---

## 3. 独立性の要件（最重要）

PAI の価値は**独立性**にある。以下を満たさない検査は PAI とは呼べない（＝ TC[DLV] の別名に堕す）。

1. **実施主体の独立**: 当該完成品を実装した Author とは**別の主体**が実施する。
   - Reviewer モード（`bootstrap/CLAUDE-reviewer.md.template`）の**別セッション**、別担当 AI、または人間。
   - 1 セッションで Author と PAI を兼務しない（AT と同じ独立性規律、`CLAUDE.md` ハードルール）。
2. **ケース源は SPEC（最重要）**: 検査ケースは **SPEC（人間承認済みの Why/What）から作成する**。
   - **下流チェーン（UC / DD / TS / TC）も実装コードも in-repo の TC も見ずに**、**SPEC-REQ から独立に導出**する。
     SPEC が真であり、CTR（凍結境界契約）は SPEC を外部公開する**表面＝検査の経路**として使う（SPEC を完成品の
     観測可能な振る舞いに翻訳する地点）。
   - **なぜ SPEC 由来か**: PAI は「SPEC→UC→…→SRC→完成品」のパイプライン全体が **SPEC の意図を完成品まで忠実に
     届けたか**を検査する。テストを SPEC から独立に起こすからこそ、**チェーンのどこか（または SCP プロセス自体）で
     SPEC 意図が失われた地点が露見する**（§1.1 のプロセス穴検出はこの独立性に由来する）。下流成果物や実装からテストを
     起こすと、意図を落としたのと同じ盲点を共有し、穴を見逃す（写経は盲点の継承）。
3. **全数 mapping（P-3）**: 各 **SPEC-REQ**（および完成品サーフェスを規定する **CTR 契約項目**）に対して
   **≥1 件の PAI ケース**を割り当てる。未カバーの REQ・契約項目があれば**検査不備**として記録する
   （自己申告でなく ID 突合で機械検証する）。
4. **黒箱**: 内部実装・内部状態を参照せず、**外部から観測可能な振る舞い**のみで判定する
   （終了コード・stdout/stderr・出力スキーマ・ファイル副作用・プロトコル応答など）。

---

## 4. TC[DLV] / AT との役割分担

PAI は TC[DLV] や AT の**代替ではなく補完**である。三者は独立した別経路として並置する。

| 観点 | TC[DLV]（`12-delivery-layer.md`） | AT（`07-at-and-nfr.md`） | **PAI（本章）** |
|---|---|---|---|
| 検証軸 | 契約適合の**回帰防止** | UX・暗黙知・前提不一致 | 契約/機能の**完成品適合の発見** |
| 実施主体 | Author（in-chain） | 人間 | **Author と独立**（Reviewer/人間） |
| 箱 | 白箱・hermetic | 人間操作 | **黒箱・実成果物・実環境** |
| ケース源 | TS[DLV]（自作） | ドメイン観察 | **SPEC から独立導出（CTR は表面）** |
| 頻度 | 毎コミット（CI ゲート） | マイルストーン | **リリース前（必須）** |
| 主目的 | **既知**項目の回帰検出 | 分布外 UX 観察 | **未知**ギャップの発見 |

> 要点: TC[DLV] が「既に知っている契約をこれからも守り続ける」ための回帰ネットなのに対し、PAI は
> 「自分が見落とした契約違反を、独立な目で見つける」発見チャネルである。**両者を混同して TC[DLV] だけにすると、
> legixy の 54 件が再来する。**

## 5. 実施フローと失敗時の扱い

```
1. リリース候補をビルド（完成品を生成。デバッグ用 stub でなく実配布形態）。
2. 検査計画を起こす（PAI 本体）:
   SPEC-REQ を独立に読み、SPEC-REQ（+ それを公開する CTR 契約項目）→ PAI ケースへ全数 mapping。
   ※ 下流チェーン（UC/DD/TS/TC）・実装コード・in-repo TC は見ない（独立性 §3-2/3-3）。
3. 黒箱で完成品を実環境で実行し、SPEC 意図を満たすかを PASS / 要改善（指摘ID）/ 未実施 で記録。
4. 失敗（要改善）の二段の戻し:
   (a) 症状の修正:
       - SPEC は正しく実装が負け  → /defect-fix（`10-modification-events.md`）
       - SPEC 自体に欠陥          → /spec-change（人間承認）
   (b) プロセスの穴の点検（本質、§1.1）:
       「なぜこのクラスの不適合が全チェーン GREEN を通過したか」を root-cause 分析し、
       上流で止められる仕組み（新ゲート・新検査・新観点・新成果物型）が要るなら SCP 改善へ。
   ※ 指摘を in-repo スイートに「黙って吸収」してはならない（§6）。
5. 要改善ゼロでリリースゲート通過（`08-gates.md` §12）。
```

- **AT との振り分けの違い**: AT 失敗は「仕様の不完全性の発見」として **GAP[UC]/[SPEC]** に変換される
  （`07-at-and-nfr.md`）。PAI 失敗は **SPEC 意図に対する完成品の不適合**なので、原則 **/defect-fix**
  （SPEC は正、実装が負け）。SPEC 側の欠陥が原因なら **/spec-change**。
- **[Recurrence]（二層）**: (1) PAI で見つかった**決定論的・ONNX 非依存**なケースは、再発防止として TC[DLV]（in-repo
  回帰ゲート）へ昇格してよい（§6 の独立性を壊さない範囲で）。(2) チェーンをすり抜けた事実そのものは**プロセスの穴**の
  証拠として扱い、`ai-antipattern.md` / `perspectives.md` への昇格や SCP 章の改訂で塞ぐ（§1.1）。

## 6. in-repo 化の扱い（独立性を壊さないこと）

PAI の決定論サブセットを in-repo の TC[DLV] に取り込むこと（回帰ゲート化）は**推奨**だが、**PAI 本体（独立黒箱検査）
そのものを in-repo に畳んではならない**。

- 「PAI で見つかった項目を TC[DLV] に追加」＝**過去に見つけた違反の再発防止**（正しい運用）。
- 「PAI を廃して全部 in-repo TC[DLV] にする」＝**独立性＝発見力の喪失**（誤った運用）。Author が契約テストも書く
  ことになり、盲点を共有して新規違反を見逃す。legixy で「外部スイートしか無かった」のとは逆方向の同じ過ち。

> `12-delivery-layer.md` の「外部テストハーネスを in-repo 化」は、上記の**回帰サブセットの in-repo 化**を指す。
> 完全な独立黒箱検査は PAI として別チャネルで維持する（同章の該当注記を参照）。

---

## 7. リリースゲート

- **リリース前に PAI pass（要改善ゼロ）が必須**。`08-gates.md` §12 リリースゲートの構成要素。
- **Author 自己 Approve 不可**。判定は人間または Author と独立な Reviewer に限る（`08-gates.md` の Approve 権限規律、
  ハードルール 11 と整合）。
- PAI を実施せずにリリースする場合は、AT スキップと同格の例外として ADR に記録する（省略の既定許容はしない）。

## 8. PAI の構造（テンプレ: `templates/PAI-template.md`）

```markdown
Document ID: PAI-<AREA>-NNN
**対象成果物**: <完成品名（例: legixy 実行ファイル / ts-mcp）>
**対象契約 / SPEC**: <CTR-<AREA>-NNN, SPEC-<AREA>-NNN, ...>
**実施主体**: <Reviewer セッション / 人間。Author と独立であること>
**実施環境**: <ビルド種別・モデル・フィクスチャ・OS>

# PAI-<AREA>-NNN: <タイトル>

## 1. 検査計画（契約 → ケース 全数 mapping）
| 契約項目 / SPEC-REQ | PAI ケース ID | 期待（黒箱観測） |
|---|---|---|
| ... | ... | ... |

## 2. 結果
| ケース ID | 判定 | 観測 | 振り分け（/defect-fix・/spec-change・—） |
|---|---|---|---|
| ... | PASS / 要改善 / 未実施 | ... | ... |

## 3. サマリ
PASS: N / 要改善: M / 未実施: K。要改善ゼロでリリース可。

## 4. 未カバー契約項目（検査不備）
（全数 mapping で漏れた契約項目・REQ があれば列挙。ゼロが原則）
```

## 9. ハードルール再掲（PAI 関連）

- 完成品のリリース前に、**Author と独立な主体による黒箱の完成品適合検査（PAI）**を実施し、要改善ゼロを確認する。
- **PAI ケースは SPEC から独立に作成する**（下流チェーン・実装・in-repo TC を見ない。CTR は表面、§3-2）。
- PAI を TC[DLV] に置換しない（独立性＝発見力を保つ、§6）。
- PAI 失敗は二段で戻す: 症状＝/defect-fix（SPEC 正・実装負け）or /spec-change（SPEC 欠陥）、**構造＝プロセスの穴の
  point-check と SCP 改善**（§1.1/§5）。PAI は完成品検査であると同時に、**プロセスが自分の盲点を発見する経路**である。
