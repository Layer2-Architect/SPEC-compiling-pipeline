# SCP テストベッド運用ガイド

このドキュメントは **SCP をテストベッドとして運用** する目的と手順を示す。SCP は SCP の実験枝であり、本番適用前に挙動・効果・コストを実検証するための系統。本番適用は SCP 本体に戻ってから行う。

## 1. テストベッドとしての位置付け

| 項目 | SCP | SCP |
|---|---|---|
| 系統 | 本番 | 実験枝（テストベッド） |
| 前段ループ | レンズ文書（§2）でのみ言及 | プロセス本体に統合（`03a-frontend-pass.md`、ハードルール 9） |
| 新規 typecode | なし | QSET, SPP, FCR |
| Claude Code に対する強制力 | レンズのみ（実質的には未採用） | CLAUDE.md ハードルール + trace-check.sh ゲート |
| 用途 | 既存プロジェクトの本番運用 | 前段ループの効果検証・コスト測定・問題発見 |
| 改廃判断 | テストベッド結果に応じて受け入れ可否を決定 | 検証用、随時改廃 |

**重要**: SCP 側を変更してはならない。テストベッドの目的は「SCP 側に何を逆輸入するかを決めるための実検証」であって、両者の同期ではない。

## 2. テストベッドの目的

検証すべき問いは具体的に:

1. **Claude Code は本当にハードルール 9 を遵守するか**
   - 新規 SPEC を渡したとき、最初に行うのは前段ループ起動か?
   - TP/UC 着手指示を出したとき、FCR.frontend_status を確認するか?
   - `bash scripts/trace-check.sh` をフェーズ完了主張前に実行するか?
2. **前段ループの反復は何回程度で収束するか**
   - 業務系 SPEC、技術系 SPEC、UI 系 SPEC で反復回数の傾向は変わるか?
   - 反復回数が多すぎる SPEC はどんな性質を持つか?
3. **QSET の質問品質は十分か**
   - AI が生成する質問は「後段コンパイル必須項目に結びついている」か?
   - 雑談質問・設計判断丸投げの質問が混入していないか?
   - Adversary が質問の妥当性を有効に監査しているか?
4. **SPP の差分案は適切か**
   - 開発者の意図を正確に反映しているか?
   - 設計判断が混入していないか?
   - 代替案の併記は実際の判断補助になっているか?
5. **検出されない不足が残るか**
   - 前段 ACCEPTED で TP[SPEC] に進んだ後、TP/GAP が「前段で検出されるべきだった不足」を捕捉する頻度は?
   - 捕捉された場合、前段検査項目に追加すべきか?
6. **コストはどの程度か**
   - LLM トークン消費（QSET 生成、SPP 生成、FCR 発行）
   - 開発者時間（QSET 回答、SPP 承認）
   - SPEC 確定までの所要時間（前段ループ込み）
7. **ハードルール 9 のスキップ ADR は累積するか**
   - スキップが常態化していないか?
   - スキップ理由のパターンは何か?

これらの問いに対する経験的データを集めることがテストベッドの責務。

## 3. 運用手順

### 3.1 セットアップ

```bash
# 既存 SCP プロジェクトとは別に、テスト用プロジェクトを用意する
mkdir -p ~/work/scp-testbed
cd ~/work/scp-testbed
git init

# SCP を docs/ 配下に配置
mkdir -p docs
cp -r /path/to/SCP docs/SCP

# 初期化
bash docs/SCP/bootstrap/init-tree.sh

# AREA コード、プロジェクト名を置換
sed -i 's/<YOUR-AREA>/TB1/g; s/<your-project>/scp-testbed/g' .trace-engine.toml
sed -i 's/<AREA>/TB1/g' CLAUDE.md
```

### 3.2 1 つの SPEC を流す

```bash
# 1. SPEC を書く（templates/SPEC-template.md を参考に）
cp docs/SCP/templates/SPEC-template.md docs/specs/SPEC-TB1-001_<topic>.md
# 編集...

# 2. graph.toml に登録
# docs/traceability/graph.toml に [[nodes]] エントリ追加

# 3. Claude Code に SPEC を渡す
# Claude Code が CLAUDE.md を読み、ハードルール 9 と起動条件を認識して
# 前段ループを開始することを期待

# 4. 反復ごとに観察記録を取る
# - QSET-TB1-001 の質問品質（後述の Observation Log フォーマット参照）
# - 開発者の回答時間
# - SPP-TB1-001 の差分の妥当性
# - FCR-TB1-001 の frontend_status

# 5. ACCEPTED に到達したら TP[SPEC] へ進む
bash scripts/trace-check.sh  # 前段ループゲートが pass することを確認

# 6. 既存の SCP プロセス（TP/GAP/UC/RB/...）を回す
# - TP/GAP が「前段で検出されるべきだった不足」を発見した頻度を記録
```

### 3.3 観察ログ（Observation Log）

各 SPEC について以下を記録する。記録場所は `docs/testbed-logs/OL-TB1-NNN.md` 推奨:

```markdown
# OL-TB1-NNN: <SPEC タイトル> の前段ループ観察

**対象 SPEC**: SPEC-TB1-NNN
**SPEC 種別**: 業務系 / 技術系 / UI 系 / 統合系
**初稿の粒度**: 詳細 / 中程度 / 粗い
**観察期間**: YYYY-MM-DD 〜 YYYY-MM-DD

## 反復ごとの記録

### 反復 1: QSET-TB1-001 → SPP-TB1-001 → FCR-TB1-001

- AI が生成した質問数: <N>
- カテゴリ別内訳: 用語確認 X, 失敗時挙動 Y, 責務範囲 Z, ...
- 雑談質問 / 設計判断丸投げ質問: <件数>（あった場合は具体例）
- 開発者の回答所要時間: <分>
- SPP の差分件数: <N>
- SPP に開発者意図と乖離した差分: <件数>（あった場合は具体例）
- FCR.frontend_status: NEEDS_QUESTIONNAIRE / ACCEPTED
- 検出された未充足カテゴリ: ...

### 反復 2: ...

## 総評

- 反復回数: <N>
- 前段に費やした合計時間: <時間>
- 前段なしで TP/GAP に直接進んだ場合の推定追加コスト: <推定>
- ACCEPTED 後、TP/GAP が「前段で検出されるべきだった不足」を捕捉した件数: <N>
- 前段の検出項目に追加すべき新ルール: ...

## Claude Code の挙動

- 起動条件（新規 SPEC 受領時の前段起動）に従ったか: Yes / No / 部分的
- ハードルール 9 を遵守したか: Yes / No
- trace-check.sh をフェーズ完了主張前に実行したか: Yes / No
- 想定外の挙動: ...
```

### 3.4 トリアージと改廃

定期（推奨: 週次または SPEC 3〜5 件ごと）に観察ログを集約し、以下を判断:

1. **SCP 本体への昇格候補**
   - 前段ループの効果が明確（TP/GAP の補完 + 仕様起因バグの減少）
   - コストが許容範囲（開発者時間 + LLM トークン）
   - 副作用がない
2. **SCP 内で改修必要**
   - 質問品質の改善（前段検査項目の追加・削除）
   - ハードルール 9 のスキップ条件の精緻化
   - FCR テンプレートの拡張・縮小
3. **却下候補**
   - 反復が収束しない（無限ループ）
   - 開発者負担が過大
   - Claude Code が遵守できない

判断結果は `docs/testbed-logs/TRIAGE-YYYY-MM-DD.md` に記録する。

## 4. テストベッドの境界

### テストベッドが扱うこと

- 前段ループの効果検証
- 質問品質の改善
- ハードルール 9 の運用妥当性
- Claude Code の挙動観察

### テストベッドが扱わないこと

- SCP 本体の改修（テストベッド結果を受けて別途決定）
- 他フェーズの大規模変更（前段ループ以外の変更は別プロジェクトで）
- 本番プロジェクトでの全面採用（テストベッド完了後に判断）

## 5. テストベッド終了条件

以下のいずれかに達したらテストベッドを終了し、SCP 本体への逆輸入可否を判断する:

- 5〜10 件の SPEC を流して観察ログが揃った
- Claude Code の挙動傾向が安定して見えた
- ハードルール 9 の効果（仕様起因バグ削減）が定量的に評価できた
- 反復回数の収束分布が見えた

終了時のアウトプット:

- `docs/testbed-logs/FINAL-REPORT.md` — 検証結果の要約
- SCP 本体への変更提案（あれば PR / 差分ファイル）
- 却下する変更とその理由

## 6. 失敗パターンと注意点

### 6.1 テストベッドが「常態運用」になる罠

テストベッドは検証用。本番ではない。SCP を本番プロジェクトに導入してそのまま使い続けると、テストベッドとしての検証機能が失われる。期限を区切ること。

### 6.2 観察ログを取らずに進める罠

Claude Code が動いて成果物が積み上がっても、観察ログがなければ後で「効果があったか」を判断できない。各 SPEC で必ず観察ログを書く。

### 6.3 前段スキップ ADR を雑に積む罠

スキップ ADR を発行すれば trace-check.sh は pass するが、これが常態化すると前段ループの効果がなくなる。スキップ件数を月次で集計し、累積率が閾値を超えたら警報を出す（推奨閾値: 全 SPEC の 20% 超）。

### 6.4 Claude Code が「ハードルール 9 を守らない」ことを許す罠

挙動観察で「Claude Code が前段を飛ばした」が見えたら、CLAUDE.md と trace-check.sh の組み合わせを強化する。テストベッドの目的の一つはこの強制力の検証。
